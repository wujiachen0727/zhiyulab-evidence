# E6：QPS × 一致性 × 运维成本 三维决策矩阵

> 基于 E1-E5 实验数据 + 06-bench 实测 + 工程经验综合。
> 所有数据来源在表后注明，避免"凭感觉给数字"。

## 三维度定义

| 维度 | 含义 | 量化方式 |
|------|------|---------|
| QPS（吞吐） | 单位时间能处理多少次"获锁/释放"轮回 | 单 client 串行 round-trip QPS（实测）|
| 一致性 | 锁互斥被破坏的概率 / 失败模式 | 主从切换/分区/STW 下的行为（来自 E1-E5）|
| 运维成本 | 部署/监控/升级的人力时间 | 主观分级（低/中/高）+ 关键差异点 |

---

## 三方案对比表

| 维度 | Redis SETNX (单实例) | Redis Redlock (5 节点) | etcd | ZooKeeper |
|------|:--------------------:|:----------------------:|:----:|:---------:|
| **QPS（单 client 串行）** | ~2000 ([1]) | ~400 ([2]) | ~900 ([3]) | ~600 ([4]) |
| **QPS（生产并发）** | 10w-100w | 数千 | 10k 量级 | 10k 量级 |
| **持锁人崩溃 (E1)** | ❌ 必须等 TTL | ❌ 同样等 TTL | ✅ keepalive 断即 revoke | ✅ session 超时即删 |
| **续期失败/STW (E2)** | ❌ 双持风险 | ⚠️ 仍有时钟假设风险 | ✅ server 端独立计时 | ✅ session 端独立计时 |
| **主从/集群切换 (E3)** | ❌ 异步复制丢数据 | ⚠️ N 节点投票，更稳但有时钟争议 | ✅ raft 强一致 | ✅ ZAB 强一致 |
| **Fencing token** | ❌ 需自己实现 | ❌ Kleppmann 主张需要，antirez 反对 | ✅ revision 单调递增 | ✅ ephemeral 序号单调递增 |
| **抢占公平性** | ❌ 轮询 SETNX，先到先得不保证 | ❌ 多数票获胜，无队列 | ✅ Watch + Txn(CAS) | ✅ Watch 前驱节点（无惊群）|
| **运维成本** | 低（单进程，主从一行配置）| 高（5 个独立节点 + 时钟同步）| 中（raft 集群 + compact + quota）| 中-高（JVM + observer + ZAB 调优）|
| **客户端语言生态** | 极广（每语言都有）| 中（依赖 Redlock 实现库）| Go 原生，gRPC 多语言 | Java（Curator）成熟，其他语言一般 |
| **典型部署节点数** | 1 主 + 1-2 从 | 5 独立 master | 3 / 5 节点 | 3 / 5 节点 + observer |

### 数据来源
- [1] E1/E2 + 06-bench 实测（macOS docker 单节点 Redis 7-alpine，单 client 串行 SETNX+DEL，5000 次平均）= 1978 QPS
- [2] Redlock 5 节点投票（每次获锁 5 个 RTT）≈ 单 Redis QPS / 5 → 估算 ~400（**这是推演，非实测**——本环境无 5 节点 Redlock 集群；实际部署因网络拓扑差异较大）
- [3] 06-bench 实测 etcd 单节点 Txn(CAS)+Delete = 896 QPS
- [4] ZK 单 client 创建 EPHEMERAL_SEQUENTIAL+Delete 估算 ~600 QPS（**这是推演**——基于 ZK 写吞吐文档值和单 RTT 估算，非本机直接 benchmark）

> ⚠️ 所有"生产并发"列均为基于公开 benchmark 数据和经验估算，不同硬件/网络/数据规模差异巨大；这一列只用于"数量级对比"，不要拿来直接做容量规划。

## Worked Example：电商超卖场景

**业务**：库存扣减
**QPS**：日常 500，秒杀峰值 5000
**一致性要求**：库存不能扣超
**已有基础设施**：Kubernetes（etcd in-tree）+ Redis 主从

走决策树：
1. 必须保证不重复 → 右分支
2. QPS < 5k（峰值刚好踩线） → 左分支
3. 已有 etcd（k8s 自带） → **选 etcd**

**反对意见与回应**：
- "etcd 单 raft 写入 ~10k QPS，5k 也不是问题" → ✅ 同意
- "用 Redis + 业务幂等更轻" → ⚠️ 业务幂等需要订单号或自然键去重；如果业务侧已有强幂等设计，Redis SETNX 也能用，但要承认 E1/E2/E3 三场景的概率

## 反例：什么时候选 Redis 也是对的

| 场景 | 为什么 Redis 够 |
|------|----------------|
| 限流（每秒 N 次）| 不需要严格互斥，丢一两次锁可以接受 |
| 缓存重建去重 | 多个 worker 重建结果一致，丢锁最多多重建一次 |
| 后台批处理任务调度 | 业务侧本身有幂等（任务 ID 去重）|
| 简单业务的"防止重复点击" | 短期 TTL，宁可松一点也别拖延用户响应 |

**核心判断**：失败一次的代价 vs 严格一致性的运维成本。如果失败一次只是多算了一次缓存或多发了一次通知，Redis 性价比最高。

## 不是结论的结论

- **没有最好的锁**，只有"对你的约束最匹配的锁"
- **决策树不是真理**，它是把"我观察到的常见权衡"画成了图——你的约束如果跟图上的分叉不匹配，就不要硬套
- **真正难的不是选锁，是知道自己什么时候不需要锁**
