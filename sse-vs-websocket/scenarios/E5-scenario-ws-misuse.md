# E5：场景模拟——一个典型的 WebSocket 误用案例

> **类型**：场景模拟（基于工程常识构造，无需标注免责声明）
> **优先级**：4
> **关联章节**：第 1 章（你的 WebSocket 可能是个"豪华车"）
> **说明**：以下场景基于常见的项目经验构造，不指向任何具体公司或产品

---

## 场景设定

**项目**：一个企业内部运营后台
**功能**：实时展示任务队列状态、系统告警、数据更新
**团队**：3 人前端团队 + 5 人后端团队
**技术栈**：React + Go + Nginx + AWS ALB

## 场景展开

### 阶段 1：选型决策

项目启动时，需求是"服务端有更新时通知前端"。团队讨论：
- A 成员："用 WebSocket 吧，实时推送嘛"
- B 成员："SSE 好像也行？"
- 最终决策：**WebSocket**，因为"更成熟，大家都在用"

### 阶段 2：实现复杂度

- 前端：引入 `socket.io-client` 库，处理连接/重连/心跳/粘包
- 后端：引入 `gorilla/websocket`，实现升级处理、连接池、心跳
- Nginx 配置：`proxy_set_header Upgrade $http_upgrade` + `proxy_set_header Connection "upgrade"`
- ALB 配置：开启 WebSocket 支持，粘性会话（sticky sessions）

### 阶段 3：上线后的问题

1. **连接不稳定**：浏览器休眠、网络切换、ALB 60s 超时 → 频繁断连重连
2. **运维困惑**：粘性会话导致负载不均，部分实例连接数爆满
3. **调试困难**：用 wss:// 抓包复杂，出问题排查链路长
4. **成本上升**：连接数增长 → 更多实例 → 更多 AWS 账单

### 阶段 4：反思

这个场景的实际需求只有一条：**"服务端有更新时通知前端"**——纯单向。但团队选择了为"双向"能力支付了全价。

## 关键教训

1. **选型惯性**："大家都用"不是技术决策理由
2. **能力过剩**：为不需要的能力支付成本是常见的架构错误
3. **简单方案 > 成熟方案**：SSE 在这个场景下更简单（EventSource 内置重连、标准 HTTP、无粘性会话），更稳定
