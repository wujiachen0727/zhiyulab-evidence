# E7：逻辑推演——实时推送技术决策框架

> **类型**：逻辑推演
> **优先级**：3（文章核心交付物）
> **关联章节**：第 5 章（什么时候选什么——决策框架）
> **方法**：综合协议特性、生产经验、竞品研究，从四个维度推演决策规则

---

## 决策前提

以下推演基于以下前提：
1. 客户端-服务端通信（非 P2P）
2. 现代浏览器环境（支持 EventSource + HTTP/2）
3. 不考虑极端边缘场景（如卫星通信、军工级可靠性）

## 四维决策矩阵

### 维度 1：通信模式

| 模式 | 适用协议 | 理由 |
|------|---------|------|
| 服务端单向推送（通知、告警、数据更新） | SSE | 无需双向通道，SSE 的零额外开销是净收益 |
| 客户端发起请求 → 服务端持续返回 | SSE | 和 AI 流式完全一致的模式 |
| 双向实时交互（聊天、协作编辑） | WebSocket | 需要双向低延迟通道 |
| 混合模式（偶尔发指令 + 持续收推送） | SSE + POST | 关注点分离，比 WS 更清晰 |

**推演结论**：单向或"一问多答"模式占所有实时推送场景的 70-80%（估算，基于常见 SaaS 应用的功能分布）。这些场景下 SSE 天然匹配。

### 维度 2：可靠性要求

| 要求 | 适用协议 | 推演逻辑 |
|------|---------|---------|
| 消息可达性要求高（金融交易） | WebSocket + ACK | WS 有标准的消息确认机制 |
| 丢几条没关系（状态更新、进度推送） | SSE | EventSource 内置重连，断线后自动恢复 |
| 顺序保证重要（日志推送） | SSE | 基于 HTTP 流，天然保持顺序 |

**推演结论**：SSE 的自动重连对大多数"实时展示"场景已经足够。需要强 ACK 保证的才是 WS。

### 维度 3：基础设施成本

| 因素 | SSE | WebSocket | 说明 |
|------|:---:|:---------:|------|
| 连接建立 | 1 次 HTTP 请求 | 6 次握手 | SSE 建连成本低 83% |
| 连接维护 | 无额外开销 | 心跳帧（30s/次） | WS 每连接约 2 次/分钟心跳 |
| 负载均衡 | 标准 HTTP | 需粘性会话 | SSE 可用标准 HTTP LB |
| 防火墙 | 标准 HTTPS | 可能被阻断 | SSE 通过率接近 100% |
| 客户端实现 | 浏览器内置 | 需第三方库 | EventSource 一行代码 |

**推演结论**：从基础设施角度看，SSE 在几乎所有维度上都更"便宜"。

### 维度 4：性能边界

| 指标 | SSE | WebSocket | 条件 |
|------|:---:|:---------:|------|
| 单连接延迟 | ~80ms P99 | ~250ms P99 | 单向场景 |
| 吞吐量（单向） | 2.9x of WS | 1x | 同等网络条件 |
| 浏览器连接数上限（HTTP/1.1） | 6/源 | 不限 | HTTP/2 解决此限制 |
| 浏览器连接数上限（HTTP/2） | 100/源 | 不限 | SSE 多路复用 |
| 二进制传输 | 不支持 | 支持 | SSE 需 Base64 编码 |

**推演结论**：性能边界上，HTTP/2 已基本消除 SSE 的最大短板（连接数限制）。唯一硬伤是二进制传输。

## 决策流程图

```
你的应用需要实时推送吗？
  ↓ 是
需要双向通信吗？
  ↓ 是 → 需要超低延迟 (<50ms) 或 P2P？
  |         ↓ 是 → WebRTC / WebTransport
  |         ↓ 否 → WebSocket
  ↓ 否（单向即可）
需要传输二进制数据吗？
  ↓ 是 → WebSocket
  ↓ 否 → 浏览器环境支持 EventSource 吗？
         ↓ 是 → SSE（推荐）
         ↓ 否 → 长轮询 / HTTP Streaming
```

## 核心判断

**大多数实时推送场景不需要 WebSocket。这不是技术判断——是成本账。**

WebSocket 提供的是"双向 + 低延迟 + 二进制"的能力集。如果你的场景只需要其中一部分（大部分情况只需要"单向 + 低延迟"），你就在为不需要的能力付费。

SSE 不是 WebSocket 的"替代品"——它是不同场景的"正确选择"。
