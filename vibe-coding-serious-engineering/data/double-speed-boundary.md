# 双速工程边界推演

## 前提

1. 探索期的主要目标是缩短反馈周期。
2. 交付期的主要目标是让变更可审查、可回滚、可追责。
3. 同一套标准无法同时最大化“探索速度”和“交付责任”。

## 三步推演

| 步骤 | 推演 | 结论 |
|---|---|---|
| 1 | 如果探索期强制使用完整生产护栏，反馈速度会被过早流程化。 | Vibe Coding 在探索期有合理价值。 |
| 2 | 如果交付期沿用探索期标准，代码虽然更快进入主干，但缺少可追责证据。 | 风险不是生成代码，而是放行标准被降低。 |
| 3 | 因此有效团队需要阶段切换点：从“候选实现”进入“可交付变更”时补齐护栏。 | 双速工程不是折中，而是分阶段优化不同目标。 |

## 论证用途

这是一条逻辑推演，不是实测数据。它用于支撑第三章“双速工程”的理论边界：探索期要快，交付期要慢下来补证据。
