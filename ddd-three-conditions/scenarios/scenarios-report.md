# 场景模拟：何时浪费 + 何时值回

> 实验时间：2026-05-25

## E4：浪费成本场景 — 纯 CRUD 时 DDD 空转

### 场景设定

后台管理系统的"标签管理"模块：增、删、改、查。
- 标签只有两个字段：name + color
- 没有业务规则（不需要校验标签名唯一性、不需要级联删除）
- 典型的"贫血模型"——Entity 只有 getter/setter

### 简单三层实现

```
handler.go:  CreateTag / UpdateTag / DeleteTag / ListTags  (~40行)
service.go:  直接转调 repo（因为没有业务规则）             (~30行)
repo.go:     CRUD SQL                                      (~50行)
```

**总计：3 文件，~120 行**

### DDD 实现（按规范走）

```
controller.go:      路由分发                               (~30行)
dto.go:             CreateTagCommand, UpdateTagCommand,
                    TagResult, Assembler                    (~40行)
app_service.go:     编排（但实际只是透传）                  (~35行)
domain_service.go:  空壳（无业务规则可放）                   (~15行)
tag_entity.go:      只有 ID/Name/Color getter               (~25行)
tag_repository.go:  接口定义                               (~15行)
tag_repo_mysql.go:  实际 CRUD                              (~50行)
```

**总计：7 文件，~210 行**

### 空转分析

| 文件 | 实际贡献 | 评价 |
|------|---------|------|
| dto.go | 结构体转来转去 | 纯仪式，字段 1:1 映射 |
| app_service.go | 调一下 domain_service | 透传层，删掉没影响 |
| domain_service.go | 调一下 repo | 空壳，没有业务逻辑 |
| tag_entity.go | 存字段 | 贫血模型，和结构体无异 |
| tag_repository.go | 定义接口 | 唯一实现，接口多余 |

**5 个文件是"仪式性存在"**——它们的代码在运行时只是数据透传，不包含任何判断、校验或编排逻辑。

### 量化空转成本

- 每个 CRUD 接口多写 ~90 行"仪式代码"（210 - 120 = 90）
- 一个典型后台管理系统有 15-20 个类似模块
- **总空转成本**：15 × 90 = **1350 行仪式代码**
- 这些代码需要维护、需要 code review、需要写测试——但不产生任何业务价值

## E3：值回成本场景 — 业务变更时 DDD 优势

### 场景设定

用户注册模块上线 3 个月后，新需求：
1. 增加手机号注册（可选，与邮箱二选一）
2. 增加邀请码逻辑（有邀请码的用户获得额外权限）
3. 注册成功后除了发邮件，还要发站内通知

### 简单三层的改动

```
handler.go:    新增 Phone/InviteCode 字段                      (+5行)
service.go:    Register 函数改动：
               - 新增手机号校验逻辑                             (+15行)
               - 新增邀请码查验逻辑                             (+20行)
               - 新增站内通知调用                               (+5行)
               - 修改原有校验逻辑（邮箱变可选）                  (改~10行)
repo.go:       新增 InviteCodeExists 查询                       (+10行)
               修改 CreateUser 新增字段                          (改~5行)
```

**改动：3 文件全改，service.go 变得臃肿（一个函数 80+ 行）**

痛点：
- `Register` 函数从 30 行膨胀到 80+ 行
- 校验逻辑、持久化逻辑、通知逻辑全在一个函数里
- 下次再加需求？继续膨胀

### DDD 版的改动

```
valueobject/phone.go:       新增 Phone 值对象（含校验）          (新文件 +25行)
domain_service.go:          注册逻辑调整：
                            - 邮箱/手机二选一规则                  (+10行)
                            - 邀请码校验（领域规则）               (+15行)
entity/user.go:             新增 Phone 字段 + InviteCode 关联      (+10行)
repository interface:       新增 InviteCodeExists 方法             (+3行)
repository impl:            实现新方法                             (+10行)
event:                      新增 UserRegisteredWithInvite 事件      (+8行)
event_handler:              新增站内通知 handler                    (+15行)
dto.go:                     新增字段                               (+5行)
```

**改动：分散在 5-6 个文件，但每个文件改动量小（≤25行），且改动高度局部化**

优势：
- 校验逻辑封装在值对象中——手机号校验独立于邮箱校验，不会相互影响
- 新通知通过事件机制添加——不修改注册主流程
- 邀请码逻辑在 domain service 中——不和 HTTP 层纠缠
- 每个改动点的**影响半径**可控

### 量化对比

| 维度 | 简单三层 | DDD |
|------|:-------:|:---:|
| 改动文件数 | 3 | 5-6 |
| 单文件最大改动 | 50+ 行（service.go） | 25 行 |
| 引入 bug 风险 | 高（大函数内多处修改） | 低（改动隔离） |
| 测试影响范围 | 重测整个 Register 函数 | 只测新增值对象 + 改动的 service 方法 |
| 函数复杂度变化 | Register: 30行→80行 | 各方法保持 ≤30行 |

### 关键洞察

DDD 的价值不在"改动文件数更少"——实际上改动文件数更多。
DDD 的价值在"**每次改动的影响半径更小，bug 传播路径更短**"。

当业务变更频率高时（每月 2-3 次需求迭代），这种隔离性的价值会累积。

## E5：Break-even Point 推演

### 假设前提

- 初始开发成本差异：DDD 多投入 ~115 行代码（实验数据）
- 按时间估算：DDD 初始多花 ~2-3 小时（含设计思考）
- 每次变更：简单三层平均改动 50 行/大函数膨胀，DDD 平均改动分散在 25 行/多个小文件

### 成本模型

```
C_init = DDD 初始额外投入 ≈ 2.5 小时
C_save_per_change = 简单版变更成本 - DDD版变更成本

简单版单次变更成本 ≈ 1.5 小时（改大函数 + 回归测试整个函数）
DDD版单次变更成本 ≈ 1.0 小时（改小模块 + 只测改动部分）
C_save_per_change ≈ 0.5 小时

Break-even = C_init / C_save_per_change = 2.5 / 0.5 = 5 次变更
```

### 结论

- **5 次业务变更后，DDD 的初始投入回本**
- 但这个推演基于"每次变更都命中 DDD 的优势场景"（非 CRUD 类变更）
- 如果变更中 50% 是简单 CRUD 修改（不触及领域逻辑），则 break-even 约 **10 次变更**

### 边界条件

| 场景 | 预估变更频率 | Break-even 时间 | 建议 |
|------|:-----------:|:--------------:|------|
| 核心业务模块（订单/支付） | 月均 3-4 次 | 2-3 个月 | ✅ 值得 DDD |
| 中等模块（用户管理） | 月均 1-2 次 | 5-10 个月 | ⚠️ 视团队能力 |
| 低频模块（后台配置） | 季度 1 次 | 2.5-5 年 | ❌ 不值得 |
| 纯 CRUD 模块 | 几乎不变 | ♾️ 永远回不了本 | ❌ 明确浪费 |

## E6：团队能力门槛（逻辑推演）

### 前提

DDD 不是个人项目——它是一个**团队级**的架构决策。

### 推演链

1. DDD 要求团队成员理解一套概念体系（聚合根、值对象、领域服务、端口-适配器...）
2. 理解概念 ≠ 正确使用——"什么放值对象、什么放领域服务"需要反复讨论
3. 讨论成本与团队规模成正比——3 人团队每次讨论 30 分钟，8 人团队每次讨论 1.5 小时
4. 如果团队中 DDD 经验不足的人 > 50%，决策质量下降 → 产出"形式 DDD"（有分层但逻辑还是意面式）

### 信号判断

满足 ≥ 3 个信号 → DDD 可能值得投入：

1. **团队 ≥ 3 人有 DDD 实践经验**（不是看过书，是做过项目）
2. **业务规则变更频率 ≥ 月均 2 次**（否则 break-even 太远）
3. **系统预期生命周期 ≥ 2 年**（否则初始投入收不回来）
4. **有领域专家可以协作**（否则 Ubiquitous Language 形同虚设）
5. **团队愿意为"概念讨论"投入时间**（每周 1-2 小时的设计讨论）

满足 < 3 个 → 建议从 strategic DDD（Bounded Context 划分 + 通用语言）入手，不上 tactical patterns。
