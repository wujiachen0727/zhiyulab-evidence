# 三层成本框架

类型：逻辑推演  
标注：[推演]  
用途：支撑 E5，把 E1/E2/E3/E6 的实测结果组织成正文主框架。

## 前提

1. Python 的设计目标之一是降低表达摩擦：语法更直接、对象模型更统一、运行时更灵活。
2. 工程系统需要的不只是“写出来”，还包括：可复现、可协作、可测试、可交付、可长期维护。
3. 如果一种语言在语法层降低了显式约束，部分约束会在运行时或工程流程中重新出现。

## 三层拆分

| 层级 | Python 提供的收益 | 复杂度转移到哪里 | 对应论据 |
|------|------------------|------------------|----------|
| 语法层 | 第一版代码更快写出，表达成本低 | 不是主要转移层；这里是真收益 | R1 + 章节解释 |
| 运行时层 | 动态类型、线程 API 等让写法更轻 | 类型错误暴露时间、GIL 并行边界、解释器实现 | E1 + E2 + R2 |
| 工程层 | 单文件脚本低摩擦启动 | 依赖、测试、类型检查、CI、README、包结构 | E3 + E4 + E6 |

## 推理链

1. 如果复杂度被语法层吸收，读者会在写第一版时少付账。
2. 但长期交付需要稳定边界，边界不会因为语法简单而自动存在。
3. 所以项目越接近多人协作和持续交付，越需要把隐性约束显式化。

## 反例边界

- 小脚本、一次性分析、探索性 notebook：复杂度没有进入协作边界，Python 的简单基本就是净收益。
- 长期服务、多人协作、需要稳定交付的项目：简单仍然有价值，但必须配套测试、类型、依赖和运行约束。
- 包管理、CI、测试不是 Python 独有问题；本文只讨论 Python “脚本低摩擦”与“工程显式约束”之间的落差。

## 可供正文引用的结论

Python 的简单最值钱的位置，是语法层和探索层。越往运行时和工程层走，越不能把“写起来轻”误认为“交付起来也轻”。这不是 Python 的失败，而是复杂度结算位置发生了变化。
