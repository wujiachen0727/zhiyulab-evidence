# 个人脚本被团队复用后的工程补账场景

## 场景定位

类型：场景模拟  
标注：基于工程常识的合理场景，不虚构具体人名、公司名或精确业务统计。  
用途：支撑 E4，说明个人脚本进入团队复用后，隐性复杂度会从“作者脑子里”转移到工程约束里。

## 场景

一个 20 行 Python 脚本最初只做一件事：读取 `invoices.csv`，把 `amount` 加总，打印结果。

第一次写它的人很舒服：

```bash
python invoice_summary.py
```

能跑，结果也对。到这里，Python 的简单就是收益。

问题从第二个人复用它开始出现。

他会问五个问题：

1. 用哪个 Python 版本？
2. 输入文件路径能不能改？
3. `amount` 为空、带小数、带货币符号时怎么处理？
4. 改完以后怎么知道没破坏旧逻辑？
5. 这东西以后是手动跑、定时跑，还是放进 CI / 服务里？

这些问题不是“挑刺”。它们只是把原来藏在作者使用习惯里的信息，搬到了协作边界上。

## 复杂度转移

| 原来在哪里 | 团队复用后转移到哪里 | 需要补的约束 |
|-----------|----------------------|--------------|
| 作者知道输入文件叫什么 | README / 参数 / 配置 | 运行说明、参数边界 |
| 作者知道机器上有什么依赖 | venv / requirements / pyproject | 依赖声明、版本约束 |
| 作者手动看输出对不对 | tests | 自动测试 |
| 作者知道哪些输入不合法 | 类型标注 / 校验逻辑 | 输入模型、异常处理 |
| 作者本地手动运行 | Makefile / CI | 可复跑命令、流水线 |

## 可供正文引用的结论

个人脚本的简单，很多时候来自“上下文没有写出来”。当它进入团队协作，上下文就必须被显式化：版本、输入、异常、测试、部署路径。复杂度没有突然增加，只是从个人记忆转移到了工程文件和流程里。
