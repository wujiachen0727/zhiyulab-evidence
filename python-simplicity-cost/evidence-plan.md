# 论据构造计划

## 核心论点

Python 的简单不是消灭复杂度，而是把复杂度从语法层转移到运行时、测试、类型约束和工程纪律里；项目越大，这笔账越显性。

## 论据需求拆解

| 子论点 | 需要证明什么 | 优先论据类型 |
|--------|--------------|--------------|
| 语法层简单是真的 | Python 的设计确实主动追求可读性、简单性和显式性 | 官方设计材料 + 逻辑推演 |
| 运行时代价也是真的 | 动态类型、GIL 等机制把一部分约束推迟到运行时或解释器实现中 | 实验验证 + 官方材料 |
| 工程层需要补账 | 项目从脚本到服务后，依赖、类型、测试、构建和部署复杂度会自然出现 | 数据实测 + 场景模拟 |
| 复杂度转移不是缺陷清单 | 这些代价与收益共生，关键是识别边界 | 逻辑推演 + 决策框架 |

## 论据优先级排序

1. **E3：脚本到项目复杂度实测** — 最贴近标题里的“工程里越贵”。
2. **E1：类型错误前移实验** — 证明“动态简单”会把部分错误留到运行期或工具补账期。
3. **E2：GIL 并发实验** — 证明“写法简单”不等于“并行模型简单”。
4. **E5：三层成本框架** — 把实验结果组织成读者能带走的认知模型。
5. **E4/E6：团队复用场景 + 交付清单计数** — 补足工程协作层面的代价。
6. **R1/R2：官方材料** — 只做事实锚点，不替代核心论据。

## 降级预案

| 风险 | 降级方案 |
|------|----------|
| 本地缺少 mypy/pyright | 优先 `pip install`；失败则用 Python 运行时错误 + 静态类型伪检查说明，不声称实测类型工具。 |
| GIL 实验噪声过大 | 不写具体性能倍数，只写“未线性提速”的本机观测，并用官方 GIL 定义兜底。 |
| 项目复杂度计数显得主观 | 明确计数口径：文件数、命令数、配置项数，不用“复杂很多”这类空泛形容。 |
| 没有作者个人 Python 经历 | 不编造经历，改用可复现小项目和场景模拟。 |
