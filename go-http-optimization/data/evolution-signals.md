# 演进信号清单

> 从 E1-E5 实验数据中提炼的量化决策阈值。读者可直接用于判断自己服务的演进阶段。

## 5 层演进信号

### 第0层→第1层：你需要开始优化了

**触发信号**（满足任一）：
- 默认配置下，100 并发 P99 > 50ms
- 服务器在 Slowloris 攻击下 200 连接即瘫痪
- 内存使用持续增长（goroutine 泄漏嫌疑）

**行动**：加上超时配置，引入 sync.Pool，用 pprof 找瓶颈

### 第1层→第2层：单机优化到头了

**触发信号**（满足 2 个以上）：
- 单机 P99 > 200ms 且 GC 优化 2 轮无改善（改善 < 10%）
- 业务复杂度到了 Layer 3+（含外部调用），QPS 衰减超过 80%
- 多个团队争抢同一个 repo 的发布窗口
- 单一故障域：一台机器挂了全挂，且 RTO > 30 分钟

**行动**：考虑模块化单体——用 Go interface + internal 包做模块边界

### 第2层→第3层：模块边界不够了

**触发信号**（满足 2 个以上）：
- 不同模块需要独立扩展（如支付模块 QPS 是其他模块 10 倍）
- 不同模块需要不同的部署节奏（如支付模块每天发，其他模块每周发）
- 团队超过 15 人，模块间代码冲突频繁
- 某个模块需要独立的 SLA（如支付 99.99% vs 内容 99.9%）

**行动**：将高扩展差异的模块拆为独立服务，其余保持进程内

### 第3层→可观测性：分布式需要眼睛

**触发信号**（满足任一）：
- 服务超过 3 个，排查问题需要登录 3+ 台机器看日志
- 一个请求跨 3+ 个服务，P99 异常但不知道卡在哪
- 月度 P0 事故中，"定位慢"是主要瓶颈

**行动**：引入分布式追踪（OpenTelemetry），统一日志格式，建立服务拓扑图

### 回退信号：你拆过头了

**触发信号**（满足 2 个以上）：
- 一个服务挂了触发所有服务的告警
- 改一个 API 需要同步改 3+ 个服务
- 首页请求产生 100+ 网络调用
- 运维人力 > 开发人力的 50%
- 团队人均维护 > 3 个服务

**行动**：合并高耦合服务为模块化单体（回迁不是失败，是演进的闭环）

## 使用方法

1. 对照信号清单，数你满足了几个
2. 如果某层的信号满足 ≥ 阈值 → 执行对应行动
3. 如果回退信号满足 → 先合并，再重新评估
4. 没有信号触发 → 你在正确的层，保持现状
