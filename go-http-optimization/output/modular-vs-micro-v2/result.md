# E3: 模块化单体 vs 微服务 延迟对比结果

[实测 Go 1.26.2 darwin/arm64]
[推演：模拟业务处理] 每个模块 1ms time.Sleep
说明：使用 HTTP 代替 gRPC，实际 gRPC 延迟更低，对比结果是保守估计

## 核心数据

50并发，5000请求：

| 指标 | 模块化单体(进程内) | 微服务(HTTP调用) | HTTP慢几倍 |
|------|-------------------|-----------------|-----------|
| P50 | 3.44ms | 4.83ms | 1.4x |
| P90 | 3.51ms | 13.66ms | 3.9x |
| P99 | 3.73ms | 31.82ms | 8.5x |

## 核心结论

- P50 差异不大（1.4x），但 P99 差异巨大（8.5x）
- 额外延迟来源：HTTP协议栈开销、TCP连接管理、JSON序列化/反序列化
- 若使用 gRPC(protobuf+HTTP2+长连接)，延迟约为 HTTP 的 1/2~1/3
- **即使用 gRPC，P99 仍会有 3-4x 的差距**

## 正文引用方式

"实测：同一业务逻辑（订单→库存→支付），进程内调用 P99=3.73ms，HTTP跨服务调用 P99=31.82ms——8.5倍差距。换 gRPC 大约是 HTTP 的 1/3，P99 差距仍在 3 倍以上。"
