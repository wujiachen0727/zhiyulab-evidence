# Go HTTP 单机天花板实验

> [实测 Go 1.26.2 darwin/arm64] | 100 并发 | 10000 请求/层

## 实验设计

5 层业务复杂度递进，每一层都是独立的 HTTP handler，运行在同一个服务器上：

| 层级 | 业务复杂度 | 模拟内容 |
|------|-----------|---------|
| Layer 0 | 纯 JSON 序列化 | 直接返回 JSON 响应 |
| Layer 1 | +DB 查询 | 2ms `time.Sleep` 模拟数据库查询 [推演：模拟延迟] |
| Layer 2 | +Redis 缓存 | 0.5ms 缓存查询 + 2ms DB 查询（缓存 miss） [推演：模拟延迟] |
| Layer 3 | +外部 HTTP 调用 | Layer 2 + 调用辅助 HTTP 服务器（3ms） |
| Layer 4 | +业务逻辑 | Layer 3 + 500 项字符串处理+排序 |

压测方式：Go 标准库并发 `http.Client`，100 并发，每层 10000 请求。

## 关键发现

1. **纯 JSON 序列化（L0）近乎 2 万 QPS**：Go 的 HTTP + JSON 序列化管线极为高效，100 并发下轻松达到 ~20K QPS。

2. **加 DB 查询（L1）衰减 31%**：2ms 的模拟 DB 延迟让 QPS 下降约三成，但 P99 反而更低（12.3ms vs 14.6ms），这是因为 sleep 延迟使请求处理时间更均匀，尾部延迟方差缩小。

3. **加缓存+DB（L2）断崖式衰减 91%**：0.5ms 缓存查询 + 2ms DB 查询的串行等待导致 QPS 从 ~14K 暴跌到 ~1.8K，P99 从 12ms 飙升到 309ms。串行 I/O 等待是 Go HTTP 性能的最大杀手。

4. **加外部 HTTP 调用（L3）再降 80%**：网络调用的连接建立、数据传输开销叠加 3ms 处理延迟，QPS 仅剩 365，P99 突破 1.5 秒。跨进程网络调用比进程内 sleep 代价高得多。

5. **加业务逻辑（L4）QPS 仅剩 162**：完整的业务栈下，单机 QPS 从 2 万跌到 162，衰减 99.2%。但这不是 Go 的问题——而是业务复杂度的固有代价。

6. **GC 停顿极短**：即使在最重的 L4 负载下，GC 停顿普遍在微秒级别（最长的 1.23ms），GCCPUFraction 仅 1.78%。Go 的低延迟 GC 确实名副其实——GC 不是性能瓶颈。

7. **真正的瓶颈是串行 I/O 等待**：从 L1 到 L3，性能断崖的根源不是 CPU 也不是 GC，而是 goroutine 在等待 I/O 时占用了并发槽位。100 并发下，当每个请求需要串行等待 5.5ms+ 的 I/O 时，理论 QPS 上限就是 ~18K（100/5.5ms），实际受连接开销影响更低。

## 实验环境

- Go 版本：1.26.2
- 平台：darwin/arm64 (Apple Silicon)
- 主服务器端口：18090
- 辅助服务器端口：18091（模拟外部调用）
- 压测参数：100 并发，10,000 请求/层
- 预热：每层 100 请求

- `[推演：模拟延迟]` — 使用 `time.Sleep` 模拟的延迟，非真实 I/O
