# sync.Pool 基准测试结果

> [实测 Go 1.26.2 darwin/arm64]

## 实验设计

- **场景**：HTTP 服务器 JSON 序列化（模拟业务响应，含嵌套结构体，~500 字节/响应）
- **对比**：每次请求创建新 `bytes.Buffer` vs `sync.Pool` 复用 buffer
- **压测参数**：100 并发，50000 总请求
- **采集方式**：`runtime.ReadMemStats`（客户端内存 + GC pause）+ `GODEBUG=gctrace=1`（服务端 GC 日志）
- **实验控制**：服务端和客户端分进程运行，连接池复用 TCP 连接，每组跑 3 轮取中位数

## 结论

1. **服务端 GC 次数减少 56%**：无 Pool 的服务端 5000 请求触发 25 次 GC，有 Pool 的仅 11 次。`sync.Pool` 复用 buffer 显著减少了短生命周期对象的分配，降低了 GC 触发频率。

2. **客户端 GC 停顿改善有限**：客户端的 GC 停顿差异不大（13.89ms vs 12.87ms，-7.4%），因为客户端的内存压力主要来自 HTTP 连接管理而非 buffer 分配。

3. **QPS 和延迟小幅改善**：有 Pool 版本 QPS 提升 ~2.5%，P90 延迟降低 ~4.6%。改善幅度不大，因为此场景中 JSON 序列化（含 makeResponse 的分配）才是主要开销，`bytes.Buffer` 只是众多分配源之一。

4. **Mallocs 差异极小**：客户端的内存分配次数几乎没有差异（2,852,439 vs 2,851,268），因为 Pool 的收益体现在服务端。

5. **关键洞察**：`sync.Pool` 的收益在**服务端**体现最明显——GC 次数减半。但在只序列化单个 ~500 字节响应的场景下，绝对收益有限。当响应更大（如 JSON 数组含数百元素）或序列化频率更高时，Pool 的收益会更加显著。

## 实验局限

- 服务端和客户端在同一台机器上运行，网络开销极低（loopback），生产环境中网络延迟会掩盖部分 GC 停顿差异
- `GODEBUG=gctrace=1` 本身会引入轻微的性能开销，因此 GC 日志在独立轮次中收集
- 响应体约 500 字节，相对较小；更大的响应体会使 Pool 的收益更突出
- Go 1.26 的 GC 已经非常高效（P99 停顿 < 5ms），sync.Pool 的边际收益在高并发+大对象场景更明显
