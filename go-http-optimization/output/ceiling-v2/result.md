# E5: Go HTTP 单机天花板衰减曲线

[实测 Go 1.26.2 darwin/arm64]
[推演：模拟IO延迟] time.Sleep模拟DB/Redis/HTTP调用

## 核心数据

100并发，8000请求/层：

| 层级 | QPS | P50 | P99 | 相对L0 |
|------|-----|-----|-----|--------|
| L0:纯JSON | 10784 | 9.12ms | 17.77ms | 100% |
| L1:+DB(2ms) | 10776 | 9.23ms | 10.65ms | 99.9% |
| L2:+Cache+DB(2.5ms) | 1775 | 43.62ms | 237.18ms | 16.5% |
| L3:+HTTP调用(5.5ms) | 1705 | 50.10ms | 219.34ms | 15.8% |
| L4:+业务逻辑(6.5ms) | 1729 | 45.63ms | 238.10ms | 16.0% |

## 关键发现

1. **L0→L1 几乎无衰减**：Go 的 goroutine 模型让 IO sleep 期间 CPU 被其他请求使用——这正是 Go 单机比 Java 线程池模型强的地方
2. **L1→L2 断崖式衰减**：QPS 从 10776 暴跌到 1775（衰减 84%）——多个串行 IO 叠加是真正的杀手
3. **L2→L4 趋于平缓**：说明"串行IO数量"是瓶颈，不是"总延迟时间"

## 核心结论

- Go 单机"百万QPS"只在纯 JSON 场景成立，真实业务复杂度下衰减 84%
- 但 Go 的 goroutine 模型让单个 IO 延迟（2ms DB）几乎不影响吞吐——这是 Go 相比 Java 的独特优势
- 真正的瓶颈是**多个串行 IO 叠加**，不是单个 IO 的延迟

## 正文引用方式

"实测：纯 JSON 序列化 QPS 10784，加上多个串行 IO（DB+Cache+HTTP）后 QPS 仅剩 1775——衰减 84%。但注意：单个 2ms 的 DB 查询几乎不影响 QPS，因为 goroutine 在 sleep 期间让出了 CPU。真正的杀手是串行 IO 叠加。"
