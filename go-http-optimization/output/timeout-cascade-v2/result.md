# E4: Context超时传播链式衰减结果

[实测 Go 1.26.2 darwin/arm64]
[推演：模拟处理延迟] DataLayer 5ms(90%) / 50ms(10%慢查询)

## 核心数据

3层调用链（Gateway→Service→DataLayer），100并发，5000请求，10%慢查询：

| 策略 | P50 | P90 | P99 | 成功率 | 超时率 |
|------|-----|-----|-----|--------|--------|
| 宽松超时(每层30ms) | 27.81ms | 36.49ms | 49.76ms | 88.0% | 12.0% |
| 递减超时(25→20→15ms) | 22.30ms | 37.28ms | 1210.97ms | 13.1% | 86.9% |
| 紧超时(每层15ms) | 31.38ms | 111.96ms | 1566.34ms | 0.0% | 100.0% |

## 关键发现

1. **宽松超时**：成功率最高（88%），P99 也最可控（49.76ms）——但慢请求不会被快速终止
2. **递减超时**：理论最优但实际最差（成功率 13.1%）——因为 Gateway 的 25ms 不够覆盖整个调用链
3. **紧超时**：完全不可用（0% 成功率）——每层 15ms 对 3 层调用链来说太紧

## 核心结论

**递减超时不是银弹**——如果对下游处理时间判断不准，递减超时反而更危险。关键原则：

1. **超时 = 下游预期处理时间 × 2 + 安全余量**
2. **上层超时必须能覆盖整条调用链的最坏情况**
3. **宁可宽松也不要误杀**——一个慢请求的代价远低于一个被误杀的正常请求

## 可靠性衰减公式

N层串联可用性 = (1 - p)^N：
- 每层 99.9% → 3层 = 99.7%（月停机 ~22分钟）
- 每层 99.5% → 3层 = 98.5%（月停机 ~11小时）
- 每层 99.0% → 3层 = 97.0%（月停机 ~22小时）
