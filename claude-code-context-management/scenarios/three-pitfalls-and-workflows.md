# 三个高频踩坑现场（经验落地）

> 这些现场不是假设场景，是真实发生的工程现象。
> **场景描述时不写假设句"如果你..."，写"那次/上周/这次会话"——具体到时间、任务、产出。**

## E1：context 用着用着就少了一半

### 踩坑现场

那次是周五下午，要在重构一个 Go 项目的 HTTP 中间件。从早上开始就跟 Claude Code 讨论方案——先定接口、再聊 trace 注入、然后是错误码归一化、再到日志结构……一路写到下午三点，已经有 30 多轮对话。

写到第 14 个 commit 的时候，让 Claude 帮忙生成一段对接前置 BFF 的代码。它生成出来的代码用的错误码格式是 `{ "error": "..." }` 这种早期方案。

可是上午 10 点的时候，我们明明定过：错误码要走 `{ "code": "ERR_XXXX", "message": "...", "trace_id": "..." }` 这一套——那是当时聊了快一个小时才定下来的方案，里面还讨论了 trace_id 的生成时机。

但 Claude 不知道。或者说，它的"上下文里"已经没有这段了。

### 现场细节

- **任务时长**：约 5 小时连续会话
- **轮数估算**：~30 轮（按"用户消息→Claude 回复→可能多轮工具调用"算）
- **token 累计估算**：每轮平均 600 tokens，30 轮 ≈ 18K tokens；加上工具调用的 tool_result（grep/read 文件常常 2-5K tokens）—— 实际累计估算 80-120K tokens
- **现象**：Claude 给出了"对的代码"——但对的是一小时前的方案，不是中午定下的最终方案

### 当时的错误归因

最直觉的反应是："Claude 怎么忘了？记忆力这么差？"

第二反应是："是不是模型上下文窗口不够？要换 200K 模型？"

但用的本来就是 200K context 的模型。

### 真实发生的事

后来去看了下，会话累计内容已经接近 autocompact 阈值。Claude Code 静默触发了一次 autocompact，把对话历史摘要了——摘要保留了"我们在重构 HTTP 中间件、聊了错误码、聊了 trace"这种结构化结论，但**摘要丢了细节**："错误码要走 `{ "code": "ERR_XXXX", ... }`"这种**具体格式的决策**被压缩成"讨论过错误码格式"。

到了下午三点，Claude 看到的"讨论过错误码格式"，但具体格式它已经不知道了。

### 反推机制

这正是 Anthropic 三级压缩中的 **Autocompact**：在 ~83.5% 阈值触发，把历史摘要化以保留 33K headroom。设计意图是"不让会话突然崩溃"，代价就是**细节会丢**——尤其是被讨论过但未写入代码的方案。

第一章「踩坑现场」段直接呈现这次经历，不用"假设"和"通常"。重点呈现"对的代码 vs 对的最新方案"的反差。

## E2：Subagent 帮你查完反而帮倒忙

### 踩坑现场

上周三，我让 Claude Code 启动一个 explore subagent，去搜整个项目里"所有引用了 deprecated API 的位置"。这种全仓库 grep + 上下文判断的任务，正适合给 subagent。

主会话里，我和 Claude 已经聊过一段：这个项目有个"白名单文件"——`config/legacy-apis.yaml`——里面列了一批"已知会用 deprecated API 但暂时不动"的模块。我说得很清楚："白名单里的不算违规。"

subagent 跑了大约 90 秒，回来给了一份报告：列了 47 处 deprecated API 调用，按文件分组、给了 line number、还做了优先级排序。

报告里前 12 处全在 `config/legacy-apis.yaml` 列出的模块里。

### 现场细节

- **subagent 类型**：explore（专门做代码搜索）
- **输入指令**：自然语言描述任务，没有显式列白名单
- **返回结果**：47 处违规位置，前 12 处是白名单内的"误报"
- **后续动作**：手动过滤，效率约 = 自己 grep 一遍，但多花了 90 秒等 subagent

### 当时的错误归因

最直觉的反应是："subagent 不够聪明，没注意到白名单。"

第二反应是："是不是 prompt 写得不够清楚？"

### 真实发生的事

我打开 Claude Code 的执行追踪一看——subagent 没有"看到"主会话里关于白名单的讨论。**它有自己独立的 context window**，启动时只接收了 Task tool 传给它的 prompt 描述，**主会话里聊过的"白名单约束"它不知道**。

不是它"忘了"，是这条信息**从来没进入它的 context**。

### 反推机制

这是 Anthropic Subagents 的核心设计：**每个 subagent 独立 context window，主会话只看到 subagent 的最终返回结果**。这个隔离设计有它的工程合理性——subagent 内部可能要 grep 几千行代码、read 十几个文件，这些 tool_result 加起来很容易 30-50K tokens。如果 subagent 不隔离，**它的探索过程会污染主会话**——把主会话的有效上下文挤掉。

但代价就是：**主会话的约束、决策、品味，subagent 默认不知道**。需要由调用者（主会话或更上层的 agent）显式传过去。

第二章「踩坑现场」段直接呈现这次经历。重点呈现"主会话讨论过 vs subagent 知道"两个事实之间的鸿沟。

## E3：Auto Memory 把你随口一说当了真

### 踩坑现场

某次写测试用例，我跟 Claude 说："这个项目暂时用 testify，等迁完 Go 1.22 再换 testing+slog。"

这是个**临时性陈述**——只针对那一个 commit 的语境。

两天后开新会话，让 Claude 帮我新建一个测试文件。它生成出来的代码上来就是 `import "github.com/stretchr/testify/assert"`——但**这个新项目根本没用过 testify，是从零开始的**。

我愣了一下，反应过来：Auto Memory 把那条"暂时用 testify"记成了我的偏好。

### 现场细节

- **触发会话**：在会话 A 中说"暂时用 testify"
- **跨会话出现**：会话 B（不同项目，2 天后）默认引入 testify
- **位置**：`~/.claude/memories/` 目录下出现了一条对应记录
- **后续清理**：手动 `claude memory edit` 删除该条 + 在 CLAUDE.md 中显式约定"测试用 std testing"

### 当时的错误归因

最直觉的反应是："Claude 自作主张推 testify，啥情况？"

第二反应是："是不是我之前哪里设了默认偏好？"

### 真实发生的事

Auto Memory 的设计意图是：让 Claude 主动记住用户的"重复偏好"——如果你在多次会话里都用 Go、都用 PostgreSQL、都不写注释……Auto Memory 就把这些当成长期事实。

代价就是：**它分不清"重复偏好" vs "临时陈述"**。"暂时用 testify"这种语句，如果 Claude 判定"用户对测试库有偏好"，就会写入 Auto Memory——下次新会话，testify 就成了"已知用户偏好"。

而且 Auto Memory 是**跨会话持久化的**——写到磁盘文件里，下次启动 Claude Code 自动加载。一旦写错，不主动清理就一直误用。

### 反推机制

这是 Anthropic Memory 设计中的 **Auto Memory 层**：让 Claude 在会话中主动判断"这是值得长期记住的偏好吗？"——是 → 写入 `~/.claude/memories/`。

工程动机：避免每次新会话都从零开始问用户"你用什么数据库？什么测试框架？"

代价：**判断不准**——把临时陈述当成偏好。

## E7：场景模拟 — 知不知道机制的工作流对照

### 第一章对照：context 爆掉

**不知道机制时**：

```
我（用户）：让我们继续重构这个中间件，加个 metric 上报。
Claude：好的，根据我们之前讨论过的错误码格式...
（生成代码用了过时的错误码格式）

我：这个错误码格式不对。
Claude：抱歉，我重新生成。我现在用的是...
（改回正确格式，但只是这一段对的，再下一段又错）
```

**知道机制后的工作流**：

```
我：（在长会话开始之前的预防）
把决策写进 CLAUDE.md，不要只在对话里说。
重要决策（如错误码格式、trace_id 规范）作为"项目硬约束"，
不依赖会话历史承载。

我：（监测累计 token）
观察到会话已经写了 ~30 轮 + 多次大文件 read，
累计估算超过 100K tokens。
主动 /clear 切新会话，把项目当前状态摘要进 CLAUDE.md。

我：（不要相信 /compact）
长会话末段如果开始出现"对的代码 vs 最新方案"对不上，
不要 /compact 继续——/compact 摘要后细节丢得更彻底。
直接 /clear 重启 + 让 CLAUDE.md 兜底。
```

### 第二章对照：Subagent 失忆

**不知道机制时的委托 prompt**：

```
"帮我搜一下整个项目里所有用了 deprecated API 的位置，
按文件分组，给个优先级排序。"
```

**知道机制后的委托 prompt**：

```
"任务：搜整个项目里所有用了 deprecated API 的位置。

约束（你不会自动知道，必须看下面这段）：
- 白名单：config/legacy-apis.yaml 里列出的模块跳过，不算违规
- 优先级标准：调用频率高 + 替代方案明确 → P0；其他 → P1
- 输出：按文件分组，每组给前 10 条

注意：你的 context window 是独立的，主会话讨论过的事情你看不到。
任何主会话约束都已写在上面。"
```

### 第三章对照：Auto Memory 误学

**不知道机制时**：

```
（自然对话）
"这个项目暂时用 testify，等迁完 Go 1.22 再换。"
→ Auto Memory 默默写入"用户偏好 testify"
→ 两天后新项目自动引入 testify
→ 用户困惑
```

**知道机制后的工作流**：

```
（临时陈述用 system prompt 区分）
"# 临时上下文（不写入 Auto Memory）
本 commit 范围：暂时用 testify
长期约束：测试库以 CLAUDE.md 为准"

（定期审 ~/.claude/memories/）
每周扫一遍：
- 哪些 memory 还成立？
- 哪些是临时陈述被误学的？
- 哪些和 CLAUDE.md 冲突的（CLAUDE.md 优先）？

（用 CLAUDE.md 兜底）
所有"长期一定不变"的偏好（语言版本、测试框架、错误处理风格）
直接写 CLAUDE.md，不让 Auto Memory 介入。
```
