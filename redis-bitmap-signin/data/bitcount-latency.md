# E5：BITCOUNT 大 key 耗时实测

## 测试方法

对每个规模 Bitmap 跑 200 次 BITCOUNT，记录平均/P50/P95/P99/max 耗时。
本地直连 Redis（无网络往返），单线程，无并发负载。

## 实测环境

Redis 8.8.0 / Go 1.26.4 / darwin/arm64 (M1)

## 数据

| Bitmap 规模 | STRLEN | MEMORY | 构造耗时 | BITCOUNT avg | P50 | P95 | P99 | max |
|------------|--------|--------|---------|-------------|-----|-----|-----|-----|
| 1KB | 1,024 B | 1,312 B | 425µs | 178µs | 172µs | 244µs | 326µs | 529µs |
| 1MB | 1,048,576 B | 1,064,992 B | 202µs | 114µs | 111µs | 136µs | 144µs | 160µs |
| 10MB | 10,485,760 B | 12,501,024 B | 643µs | 206µs | 202µs | 231µs | 239µs | 245µs |
| 100MB | 104,857,600 B | 104,874,032 B | 7.2ms | 1.41ms | 1.40ms | 1.53ms | 1.60ms | 1.61ms |
| 512MB（上限）| 536,870,912 B | 536,887,344 B | 23.3ms | 7.16ms | 7.07ms | 7.79ms | 8.83ms | 12.24ms |

## 关键发现

1. **BITCOUNT 确实是 O(N)**：从 1KB 到 512MB，耗时从 178µs 增长到 7.16ms，约 40 倍，与数据规模增长（512 倍）不成正比——Redis 内部有 SIMD 优化（popcount 指令），但趋势是线性增长。

2. **证伪了"大 key 阻塞"的强假设**：即使在 512MB（Redis Bitmap 上限）规模下，BITCOUNT P99 也只有 8.83ms，max 12.24ms。严格意义上这不算"阻塞"（通常 >50ms 或 >100ms 才算阻塞）。

3. **本地环境局限**：本测试是本地直连、单线程、无并发。生产环境有网络往返（通常 0.5-2ms）和并发负载，实际耗时会更高。按 2-5 倍放大估算，512MB Bitmap 在生产环境 BITCOUNT 可能在 15-50ms 量级。

4. **1MB 比 1KB 快的异常**：1MB Bitmap 的 BITCOUNT（114µs）比 1KB（178µs）还快。可能原因：1KB 测试受冷启动影响，1MB 测试时 CPU 缓存已预热。这是本地测试的噪声，不代表真实生产规律。

## 结论（用于正文）

- BITCOUNT 是 O(N)，但 Redis 8.8.0 的 popcount 优化让它在大 Bitmap 上仍可接受
- **512MB Bitmap 的 BITCOUNT 在本地测试中 P99 ≈ 9ms**，生产环境估算 15-50ms
- "大 key 阻塞"的说法需要修正：不是"必然阻塞"，而是"高负载下可能成为性能瓶颈"
- 真正的阻塞风险在 DEL 大 key（O(N) + 同步删除），不是 BITCOUNT
- 文章论点应从"BITCOUNT 会阻塞"调整为"BITCOUNT 在大 Bitmap 上是性能关注点，DEL 大 key 才是真阻塞"
