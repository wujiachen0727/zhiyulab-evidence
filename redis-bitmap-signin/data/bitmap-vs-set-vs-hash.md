# E6：Bitmap vs Set vs Hash 内存对比

## 测试方法

用三种数据结构存储相同的用户签到状态，对比 MEMORY USAGE。
两种场景：连续 ID（0, 1, 2...）和 稀疏 ID（0, 1000, 2000...）。

## 实测环境

Redis 8.8.0 / Go 1.26.4 / darwin/arm64 (M1)

## 数据

### 连续 ID 场景

| 场景 | Bitmap | Set | Hash | Bitmap/Set | Bitmap/Hash |
|------|--------|-----|------|:----------:|:-----------:|
| 1 万用户 | 2,112 B | 430,082 B | 480,082 B | 0.5% | 0.4% |
| 10 万用户 | 16,448 B | 4,137,586 B | 4,637,586 B | 0.4% | 0.4% |
| 100 万用户 | 131,136 B (0.13 MB) | 40,277,618 B (38.4 MB) | 45,277,618 B (43.2 MB) | 0.3% | 0.3% |

### 稀疏 ID 场景

| 场景 | Bitmap | Set | Hash | Bitmap/Set | Bitmap/Hash |
|------|--------|-----|------|:----------:|:-----------:|
| 1 万用户（步长 1000） | 1,294,416 B (1.23 MB) | 460,095 B (0.44 MB) | 510,095 B (0.49 MB) | 281% | 254% |

## 关键发现

1. **连续 ID 场景：Bitmap 完胜**。100 万用户，Bitmap 只要 0.13MB，Set/Hash 需要 38-43MB，差 300 倍。这是 Bitmap 的主战场。

2. **稀疏 ID 场景：Bitmap 反而更费内存**。1 万用户（步长 1000，max ID = 10^7），Bitmap 要 1.23MB，Set/Hash 只要 0.44-0.49MB，Bitmap 反而大 2.8 倍。

3. **证伪了"Bitmap 永远省内存"的简化判断**。Bitmap 的省内存是有条件的——连续 ID 场景完胜，稀疏 ID 场景反而更费。这正是 hash offset 反模式的根因：hash 把连续的用户 ID 打散成稀疏 offset，把 Bitmap 的优势变成了劣势。

## 结论

- Bitmap 适合连续 ID 场景（或可映射到连续 offset 的场景）
- 稀疏 ID 场景应改用 Set/Hash，或先做 ID 映射（如自增主键 → 连续 offset）
- **hash offset 是最坏选择**：既失去了 Bitmap 的省内存优势，又保留了 Bitmap 的所有缺点（单 key、无索引）
