# E1 性能 Benchmark 实测结果

**环境**：Go 1.26.2, darwin/arm64, Apple M4 Pro
**实测时间**：2026-04-12
**标注**：[实测 Go 1.26.2]

## 赋值操作对比

| 操作 | 耗时 (ns/op) | 内存分配 | 相对直接赋值的倍数 |
|------|:-----------:|:--------:|:-----------------:|
| 直接赋值 | 0.25 | 0 B/0 allocs | 1x |
| 反射赋值（FieldByName） | 85 | 0 B/0 allocs | **340x** |
| 反射赋值（缓存 Index） | 10.4 | 0 B/0 allocs | **42x** |

## 方法调用对比

| 操作 | 耗时 (ns/op) | 内存分配 | 相对直接调用的倍数 |
|------|:-----------:|:--------:|:-----------------:|
| 直接方法调用 | 0.25 | 0 B/0 allocs | 1x |
| 接口方法调用 | 0.26 | 0 B/0 allocs | ~1x |
| 反射方法调用（MethodByName + Call） | 192 | 96 B/3 allocs | **768x** |

## TypeOf / ValueOf 开销

| 操作 | 耗时 (ns/op) | 说明 |
|------|:-----------:|------|
| reflect.TypeOf | 0.26 | 几乎零成本——只是读取 interface 二元组的 type 指针 |
| reflect.ValueOf | 0.26 | 几乎零成本——包装 interface 二元组为 Value 结构 |

## 关键发现

1. **反射的开销不在"获取类型信息"，而在"按名字查找 + 动态操作"**：TypeOf/ValueOf 本身几乎免费（0.26ns），但 FieldByName 查找 + Set 操作要 85ns
2. **缓存能帮大忙但有限**：缓存 FieldIndex 后从 85ns 降到 10ns（快 8 倍），但仍是直接赋值的 42 倍
3. **方法调用是重灾区**：反射方法调用 192ns + 3 次堆分配，是直接调用的 768 倍
4. **接口调用几乎零开销**：接口调用和直接调用性能几乎一样（编译器内联），这说明"用接口替代反射"是可行的降级方案
