# E3 实验结果：团队协作合并冲突模拟

[实测 Git + Go 1.26.2]

## 实验方法

模拟 3 人团队并行修改 10 个依赖的 Go 项目：
- 开发者 A：添加新 ServiceD
- 开发者 B：添加新 ServiceE
- 开发者 C：修改 Config 添加新字段

分别在手动 DI（所有代码在 main.go）和 Wire（provider 分散在不同文件）两种方案下测试合并冲突频率。

## 结果

| 合并操作 | 手动 DI 冲突数 | Wire 冲突数 |
|---------|:------------:|:----------:|
| dev-a → master | 1 | 0 |
| dev-b → master | 0 | 0 |
| dev-c → master | 1 | 0 |
| **总计** | **2** | **0** |

## 关键发现

1. **手动 DI 的所有组装代码集中在 main.go** → 任何依赖变更都改同一个文件 → 冲突概率高
2. **Wire 的 provider 分布在不同文件** → 开发者各改各的 → 冲突概率低
3. 手动 DI 的冲突发生在 main() 函数的组装区域——代码上下文密集，解决冲突时容易引入新 bug
4. Wire 的 wire.go 虽然也可能冲突（provider 列表），但冲突范围明确且容易解决

## 可供正文引用的数据点

- "3 个人各加 1 个服务，手动 DI 出了 2 次 merge conflict，Wire 0 次"
- "手动 DI 的冲突不是'偶尔发生'——只要你改依赖图，就改 main.go，而其他人也在改 main.go"
- "Wire 让 provider 各住各的文件，冲突概率自然就下来了"
