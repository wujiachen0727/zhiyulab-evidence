# 场景模拟：Agent 自治工作流被"诚实标注"改写

> 类型：场景模拟（E4）
> 状态：基于工程常识的合理推演

## 场景：Cron Agent 凌晨自治修复 CI 红灯

### 背景设定

一个中型团队（20 人）使用 Claude Code 驱动的 CI/CD 自动修复 Agent。工作流如下：

```
CI 红灯 → Agent 自动分析失败原因 → Agent 生成修复 PR → 自动合并（如果测试全绿）
```

这是一个典型的 "Agent 自治执行" 场景：凌晨 2 点 CI 挂了，没人在线，Agent 独立决策。

### 4.7 时代的行为

```
[02:13] CI 失败：TestUserAuth timeout
[02:14] Agent 分析：数据库连接池超时，原因是连接数不足
[02:15] Agent 修复：将 max_connections 从 10 改为 50
[02:16] Agent 提交 PR，测试全绿，自动合并
[02:17] 修复完成 ✅
```

**问题**：Agent 的诊断可能是错的。timeout 可能是因为测试环境网络抖动，不是连接池。但 4.7 不会告诉你"我不确定是不是这个原因"——它直接修了，碰巧测试过了。

**隐患**：max_connections 从 10 改 50 在生产环境可能导致 OOM。

### 4.8 时代的行为

```
[02:13] CI 失败：TestUserAuth timeout
[02:14] Agent 分析：数据库连接池超时
[02:14] Agent 标注：⚠️ 不确定根因 —— timeout 可能由以下因素导致：
         1. 连接池不足（置信度：60%）
         2. 测试环境网络抖动（置信度：30%）
         3. 并发测试互相阻塞（置信度：10%）
[02:15] Agent 决策：置信度 < 80%，不自动合并，创建 Draft PR 并通知值班人
[02:15] Slack 通知：@oncall CI 失败，我生成了修复方案但不确定根因，请人工审查
```

### 工作流变化分析

| 维度 | 4.7 模式 | 4.8 模式 | 取舍 |
|------|---------|---------|------|
| 修复速度 | 4 分钟自动完成 | 需要等人介入（可能 2-8 小时） | 速度 ↓ |
| 修复正确率 | 碰运气（这次恰好测试过了） | 人机协作审查 | 正确率 ↑ |
| 隐患引入 | 可能（max_conn 改大的隐患） | 降低（人会质疑修复方案） | 安全 ↑ |
| 团队信任 | "Agent 好厉害但偶尔翻车" | "Agent 很靠谱，知道自己不确定时求助" | 信任 ↑ |

### 核心洞察

**"主动标注不确定"改写的不是一次修复，是整个自治边界的定义方式。**

过去："能修就修，修不好再说" → Agent 是全自动执行者
现在："不确定就喊人" → Agent 是带判断力的协作者

这意味着：
1. 团队需要重新定义"什么情况下 Agent 可以自主合并"
2. 置信度阈值（80%? 90%?）成为新的可配置参数
3. "诚实"不只是道德品质，是工程设计中的一个控制面

### 双面影响

**正面**：
- 减少"沉默引入"的隐患
- 提升人机协作的信任基线
- 为 Agent 自治权限提供渐进升级路径（置信度越高→自治权越大）

**负面**：
- 凌晨无人响应时 CI 持续红灯
- 团队可能因为"Agent 总是不确定"而降低对它的依赖
- 置信度校准本身是难题——模型"觉得 60%"真的是 60% 吗？
