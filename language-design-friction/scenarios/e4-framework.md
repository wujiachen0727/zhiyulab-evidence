# E4: 好摩擦力三特征判别框架

> 类型：逻辑推演 | 核心原创价值

## 推演前提

从三个实证案例（Go reflect、Rust unsafe、Java JPMS）归纳：
- Go reflect: 公认的"好摩擦力"——API 冗长但保护了性能
- Rust unsafe: 公认的"好摩擦力"——仪式感但保护了内存安全
- Java JPMS: 争议案例——意图好但社区大量弃用

## 归纳过程

### 什么是 Go reflect 和 Rust unsafe 共有、但 Java JPMS 缺少的？

| 特征 | Go reflect | Rust unsafe | Java JPMS |
|------|:----------:|:-----------:|:---------:|
| 错误路径明显变丑 | ✅ 代码变长3-5倍 | ✅ 需要 unsafe 块+注释 | ✅ 需要 module-info |
| 正确路径保持简洁 | ✅ 不用 reflect 时代码简洁 | ✅ safe Rust 零开销抽象 | ❌ 即使"正确用"也很繁琐 |
| 使用者认知提升 | ✅ 理解"为什么贵" | ✅ 理解"为什么不安全" | ⚠️ 更多是配置知识，非设计认知 |

### 推演链（3步）

1. **前提**：好的 API 摩擦力应该引导开发者走向更好的选择
2. **观察**：Go/Rust 的摩擦力让"好路径"显然更好，Java JPMS 让"所有路径"都很痛苦
3. **结论**：好摩擦力的核心不是"让坏路径难走"，而是"让好路径和坏路径的体验差距足够大"

## 三特征框架

### 特征 1：让错误路径变丑（Make Wrong Things Look Wrong）

- 错误的做法应该在视觉上、语法上明显"不对劲"
- 不需要文档告诉你"别这么做"——代码本身在喊

**检验**：
- Go reflect: 写 15 行只为读一个字段 → 明显"这不是正常做法" ✅
- Rust unsafe: `unsafe {}` 块像红色警告牌 ✅
- Java JPMS: module-info.java 是额外的配置文件 ✅（这条通过）

### 特征 2：让正确路径保持简洁（Keep Right Things Easy）

- 如果"正确做法"也同样痛苦，那摩擦力就失去了引导作用
- 关键：正确路径和错误路径之间要有明显的"体验落差"

**检验**：
- Go reflect: 泛型/接口代码 3-5 行 vs reflect 15-20 行 → 体验落差巨大 ✅
- Rust unsafe: safe Rust 零开销，编译器帮你做了大部分工作 → 体验落差巨大 ✅
- Java JPMS: 不用模块化也得写 module-info（否则框架不工作）→ 体验落差小 ❌

### 特征 3：使用者认知提升（Cognitive Upgrade）

- 摩擦力不只是"挡住你"，而是"教会你一些东西"
- 经历摩擦后，开发者应该理解"为什么这样做更好"

**检验**：
- Go reflect: 用了 reflect → 理解了运行时类型检查的成本 ✅
- Rust unsafe: 写了 SAFETY 注释 → 理解了内存安全的前提条件 ✅
- Java JPMS: 写了 module-info → 可能理解了模块隔离…但更可能只是在"满足编译器" ⚠️

## 框架应用公式

```
对目标 API/特性打分：

特征 1（错误路径变丑）    ：0-2 分
特征 2（正确路径简洁）    ：0-2 分
特征 3（认知提升）        ：0-2 分

总分 ≥ 5：好摩擦力
总分 3-4：有潜力但需改进
总分 ≤ 2：坏摩擦力（只是设计不好）
```

## 检验结果

| 案例 | 特征1 | 特征2 | 特征3 | 总分 | 判定 |
|------|:-----:|:-----:|:-----:|:----:|------|
| Go reflect | 2 | 2 | 2 | 6 | 好摩擦力 |
| Rust unsafe | 2 | 2 | 2 | 6 | 好摩擦力 |
| Java JPMS | 2 | 0 | 1 | 3 | 边界/坏摩擦 |

## 推演局限性

1. 样本量有限（3 个案例），框架可能不具备完全普适性
2. "认知提升"维度主观性较强，不同经验水平开发者感知不同
3. 框架更适合"API/语法层面的摩擦力设计"，对更高层面（架构约束）适用性待验证
