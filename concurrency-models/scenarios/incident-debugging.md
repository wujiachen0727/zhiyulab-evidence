# 场景模拟：聚合服务偶发卡住时怎么排查

## 场景

线上用户画像接口偶尔卡住。慢查询日志没有明显异常，单个下游看起来也只是偶发变慢。真正的问题是：聚合层没有清楚表达“等待到什么时候结束”和“一个下游失败后其他任务怎么办”。

## 三种排查入口

### Go / CSP 风格

先看 `context` 是否传到每个 goroutine，再看 `select` 是否监听取消信号，最后看 channel 是否有发送方或接收方泄漏。排查入口是通信结构。

### Erlang / Actor 风格

先看 parent process 的 mailbox 是否堆积，再看 worker process 是否退出、是否被 link/monitor 观察到。排查入口是进程边界和消息流。

### Java virtual thread 风格

先看虚拟线程是否被创建和等待，再看 Future/Executor 编排是否在 timeout 后取消未完成任务。排查入口仍然是调用栈和任务编排对象，只是阻塞本身不再等同于昂贵 OS 线程占用。

## 正文可用结论

并发模型影响排障时你第一眼看哪里：CSP 看通信结构，Actor 看实体边界，线程模型看调用栈和任务对象。它们不是优劣排名，而是把故障线索放在不同地方。
