# E5 场景模拟：面试 / Code Review 中的过时八股

> 类型：场景模拟。基于常见 Go 面试题和 code review 讨论构造，不包含虚构公司、虚构人名或精确统计数据。

## 场景 A：面试题

面试官问："Go slice 扩容规则是什么？"

候选人答："小于 1024 翻倍，大于 1024 增长 1.25 倍。"

这个答案在 Go 1.17 时代已经不完整，在 Go 1.18 之后更是过时。更准确的回答应该是：

1. 先看版本。Go 1.18 之后阈值从 1024 改为 256，并使用平滑过渡公式。
2. 再看元素大小。最终 cap 还会被内存分配器的 size class 对齐。
3. 所以不要只背公式，要理解 `growslice` 和 `roundupsize` 是两次决策。

## 场景 B：Code Review

review 里有人看到：

```go
buf := make([]byte, 0, 1024)
```

然后说："这个 cap 设到 1024 很关键，因为 slice 超过 1024 后只会按 1.25 倍增长。"

这个说法至少有两个问题：

- 如果团队用的是 Go 1.18+，1024 已经不是那个硬阈值。
- 即使在旧版本，最终容量也不是纯公式决定，还会被 size class 对齐。

更好的 review 不是纠结某个固定公式，而是问：

- 这个初始容量有没有来自真实数据？
- append 的上界是否可估？
- 如果不可估，当前策略在目标 Go 版本下会产生多少次分配？

这个场景用于正文收束：背公式会让人显得熟悉 runtime，但理解设计变化才是真正有用的判断力。
