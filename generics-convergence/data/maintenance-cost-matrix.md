# 维护成本矩阵

## 证据类型

- 类型：E3 数据实测 / 代理指标
- 标注：`[实测 Go 1.26.2 darwin/arm64 + 代码审计]`
- 说明：这里测量的是“新增一种可比较类型时的改动点数量、测试点数量、错误暴露阶段”。这是维护成本的代理指标，不等同于真实项目中的全部维护成本，但能说明成本会流向哪里。

## 测量方法

基于 `evidence/code/go-abstraction-paths/` 中同一去重需求，比较四种实现路径：

1. 复制粘贴：`UniqueInts` / `UniqueStrings` / 新增 `UniqueUserIDs`。
2. `any/interface{}`：保留一个 `UniqueAny`，调用侧转换。
3. 反射：保留一个 `UniqueReflect`，运行期检查 slice 和元素。
4. 泛型：保留一个 `UniqueGeneric[T comparable]`。

## 结果矩阵

| 路径 | 新增 `UserID` 支持要改几处 | 新增测试点 | 类型错误最早暴露位置 | 主要维护责任 |
|---|---:|---:|---|---|
| 复制粘贴 | 1 个新函数 + 1 组调用点 | 每种类型至少 1 组 | 编译期（函数签名），但逻辑复制易漂移 | 维护者保证多份实现一致 |
| `any/interface{}` | 通用函数 0 处，调用侧转换 N 处 | 每个调用侧都要测断言 | 运行期 panic / 类型断言失败 | 调用者保证传入和取出类型一致 |
| 反射 | 通用函数 0 处，运行期检查增加 | 需要测合法 slice、非法输入、元素可比较性 | 运行期错误 / panic 风险 | 通用函数承担运行时分支和错误处理 |
| 泛型 | 通用函数 0 处，调用侧类型自动推导 | 核心逻辑 1 组 + 关键类型少量补测 | 编译期 | 编译器承担类型一致性门禁 |

## 计数口径

- “改几处”只统计为支持新类型需要新增/修改的核心实现点，不统计文档和业务调用自然新增。
- “测试点”按最小单元测试思路估算：每种路径至少验证正常去重；运行期路径还需验证错误输入。
- 这是代理指标：真实项目成本还包括 review、调试、文档和团队认知成本，本文正文使用时应说明“这是维护成本的一个切面”。

## 可用于正文的结论

当类型种类增加时，没有泛型并不意味着成本消失，只是成本被转移：复制粘贴把成本转给维护者，`any/interface{}` 把成本转给调用者，反射把成本转给运行时和调试过程，泛型则把一部分成本提前交给编译器。
