# Java / C# / Go 泛型取舍矩阵

## 证据类型

- 类型：E7 逻辑推演 + R1/R2/R3 官方事实对照
- 标注：`[推演 + 官方资料对照]`
- 用途：避免把三门语言写成优劣排名，而是写成“历史包袱 × 运行时模型 × 兼容承诺”的不同取舍。

## 对照矩阵

| 语言 | 泛型形态 | 主要保护的东西 | 代价 | 正文表达边界 |
|---|---|---|---|---|
| Java | 编译期泛型 + 类型擦除 | 既有字节码生态和向后兼容 | 运行时泛型信息受限，部分场景需要桥接/强转理解 | 不写成“Java 泛型低级”，而写成“兼容性塑形” |
| C# | 泛型进入运行时，值类型可避免装箱 | 类型安全、性能、运行时表达力 | 运行时和 CLR 支持更重 | 不写成“C# 才是标准答案”，而写成“运行时模型允许更彻底” |
| Go | Go 1.18 后加入类型参数，强调保持 Go 味 | 简洁性、可读性、工具链稳定 | 语言规则和编译器复杂度上升，发布说明提到编译速度成本 | 不写成“Go 补课”，而写成“后发语言更谨慎地支付复杂度” |

## 可用于正文的结论

三门语言都在处理同一个压力：同一逻辑服务多种类型。但它们不是在同一张白纸上作答。Java 带着历史生态，C# 有运行时支持，Go 长期坚持简洁性。泛型形态不同，说明“加泛型”不是单一答案，而是一组被历史和运行时模型塑造的工程取舍。
