# 泛型使用边界判断

## 证据类型

- 类型：E8 逻辑推演（补充自造论据）
- 标注：`[推演]`
- 用途：防止正文把泛型写成银弹，补足“门禁不是魔法”的边界判断。

## 前提

泛型值得引入的前提，不是“能抽象就抽象”，而是同时出现以下压力：

1. 同一逻辑正在服务多种具体类型。
2. 这些类型需要保留静态类型差异，不能简单压成同一种基础类型。
3. 复制粘贴或 `any/interface{}` 已经把维护责任推给人或运行时。

## 判断表

| 场景 | 是否适合泛型 | 理由 |
|---|:---:|---|
| 只有一个类型使用一次 | 否 | 没有复用压力，泛型只增加认知成本 |
| 两三种类型共享同一算法，且类型都可比较/可约束 | 是 | 泛型能同时保留复用和编译期检查 |
| 类型差异来自业务语义，但底层表示相同 | 视情况 | 若必须防止 `UserID`/`OrderID` 混用，泛型或具名类型有价值 |
| 逻辑依赖运行时字段名、标签或动态结构 | 不一定 | 反射或代码生成可能更直接，泛型不是替代所有动态需求的工具 |
| API 边界接受外部 JSON / DB 结果 | 不直接 | 边界层仍需解析和校验，泛型主要适合边界之后的内部抽象 |

## 推演结论

泛型像门禁，不像万能钥匙。门禁适合拦“类型不该混进来”的错误；如果问题本身是动态输入、协议解析或运行时结构变化，泛型不应该硬上。这个边界能让文章更克制，也更符合止语Lab的品牌调性。
