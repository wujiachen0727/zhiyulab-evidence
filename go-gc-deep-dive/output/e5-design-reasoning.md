# E5：为什么 Go 不分代 — 设计推理链

## 推理前提（🟢已验证 — 来源：Rick Hudson ISMM 2018 演讲）

Go GC 的设计目标：**最大化 mutator 吞吐量**（应用代码获得的 CPU 时间比例）。

## 逻辑链

### 第一步：分代 GC 的理论收益

分代假说（Generational Hypothesis）：大多数对象朝生夕死。
- 新生代 GC 只扫描部分堆，频率高但代价小
- 老年代 GC 扫描全堆，频率低
- 理论收益：减少 GC 总工作量

### 第二步：分代 GC 的代价

分代 GC 必须跟踪跨代引用（老年代 → 新生代的指针）。
- 每次写操作都可能创建跨代引用
- 因此需要写屏障（write barrier）记录所有指针写入
- 写屏障开销 = 额外指令 + 内存写 → 影响 mutator 吞吐

### 第三步：Go 的设计推理

Go 团队的推理（Hudson 演讲原文）：

1. **写屏障是"全员税"**：只要开了分代，写屏障就作用于所有写操作，不论 GC 是否在运行
2. **Go 程序的延迟瓶颈在 STW**：早期 Go GC 的核心问题是 STW 时长，不是 GC CPU 开销
3. **因此选择**：不加额外写屏障 → 不分代 → 全堆并发标记
4. **代价**：每次 GC 都要扫描全堆，GC CPU 开销比分代高

### 第四步：这个推理的隐含前提

**前提**：GC CPU 开销不是延迟瓶颈。

**Green Tea GC 暗示这个前提在动摇**：
- 当堆足够大（>10GB），标记阶段扫描全堆的 CPU 开销可达 20%+
- 这时候 GC CPU 开销本身成了延迟瓶颈
- Green Tea GC 的页级扫描是对"全堆扫描"的优化，不是对分代的替代

### 第五步：为什么 Go 仍然不分代（2026 年视角）

1. **写屏障开销仍然不可接受**：Go 的并发标记只需要 Dijkstra/Yuasa 写屏障在 GC 激活时启用，分代需要永久启用
2. **Go 社区的实际痛点**：P99 尾部延迟 > GC 效率 → 验证了 Go 的选择
3. **Green Tea 的路径**：优化扫描效率而非改架构 → 更安全、更渐进

## 设计哲学结论

Go GC 的"不分代"不是技术限制，是工程选择：
- **用 GC 效率换 mutator 吞吐**
- **用内存换延迟确定性**
- **用渐进优化换架构稳定性**

这三个"用 X 换 Y"构成了 Go GC 的设计哲学核心。
