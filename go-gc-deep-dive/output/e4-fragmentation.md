# E4：碎片化场景实测 — 实测数据

> 🟢已验证 — 本机实测，Go 1.26.2 darwin/arm64

## 实测结果

**场景**：100 万个 64B 小对象，隔一释放，再分配 4KB 大对象

| 阶段 | HeapAlloc | HeapSys | 碎片率 |
|------|-----------|---------|:------:|
| 初始 | 239KB | 7.7MB | 97.0% |
| 分配 100 万个 64B 对象后 | 91.9MB | 99.4MB | 7.6% |
| 隔一释放后 | 293KB | 99.4MB | **99.7%** |
| 再分配 4KB 大对象后 | 293KB | 99.4MB | **99.7%** |

## 数据解读

1. **分配阶段碎片率低**（7.6%）：所有 64B 对象分配到对应的 size class span，利用率高
2. **隔一释放后碎片率飙升**（99.7%）：HeapAlloc 只有 293KB，但 HeapSys 仍然是 99.4MB。99.4MB 的内存被 Go runtime 从系统申请了，但 99.7% 是空闲的
3. **4KB 大对象无法复用碎片**：4KB 对象属于不同的 size class，无法放入 64B span 的空洞

## 关键结论

**碎片化是真实存在的**：释放了约 50 万个 64B 对象后，Go runtime 持有的 99.4MB 内存中只有 293KB 在用。99%+ 的碎片率验证了 chyshkala 的观点：Go 不做内存整理，碎片化在大堆场景下是结构性代价。

**注意**：Go runtime 不会立即把空闲内存还给操作系统。短期内 HeapSys 不下降不代表"永久碎片化"——runtime 会在一段时间后逐步归还。但在持续分配不同 size class 对象的场景下，碎片化确实不可解。
