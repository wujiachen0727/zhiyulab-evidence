## E1：GC Trace 实测（🟢）

**环境**：Go 1.26.2, 14P, GOGC=100

| 指标 | 值 |
|------|---|
| STW 初始暂停 | 0.025-0.37ms |
| 并发标记耗时 | 0.18-2.8ms |
| STW 结束暂停 | 0.002-0.75ms |
| GC CPU 占比 | 0-2% |

**结论**：现代 Go 的 STW 确实在亚毫秒级别。

## E2：GOGC 影响对比（🟢）

**场景**：10MB 存活堆 + 1000 万次 64B 分配

| GOGC | GC 次数 | GC CPU% | 总耗时 |
|------|:-------:|:-------:|:------:|
| 50 | 4 | 5.3% | 4.8ms |
| 100 | 2 | 4.8% | 3.4ms |
| 200 | 1 | 4.4% | 2.6ms |
| 400 | 0 | 0.0% | 2.6ms |
| off | 0 | 0.0% | 2.4ms |

**结论**：GOGC=50 比 GOGC=200 多跑 3 次 GC，CPU 开销从 4.4% 升到 5.3%，耗时多 85%。

## E3：GOMEMLIMIT + GOGC 组合（🟢）

**场景**：逐步分配 20MB，观察 GC 触发

| 配置 | GC 次数 | 堆到 18MB 时 NumGC |
|------|:-------:|:-----------------:|
| GOGC=200 | 1 | 1 |
| GOGC=200 + GOMEMLIMIT=32MB | 1 | 1 |
| GOGC=off + GOMEMLIMIT=32MB | 0 | 0 |

**结论**：GOGC=off 时 GC 完全不主动运行，GOMEMLIMIT 是安全网。

## E4：碎片化场景（🟢）

**场景**：100 万个 64B 小对象，隔一释放

| 阶段 | HeapAlloc | HeapSys | 碎片率 |
|------|-----------|---------|:------:|
| 初始 | 239KB | 7.7MB | 97.0% |
| 分配后 | 91.9MB | 99.4MB | 7.6% |
| 隔一释放后 | 293KB | 99.4MB | **99.7%** |
| 4KB 大对象分配后 | 293KB | 99.4MB | **99.7%** |

**结论**：碎片化真实存在。释放 50 万个 64B 对象后，99.4MB 内存中只有 293KB 在用。
