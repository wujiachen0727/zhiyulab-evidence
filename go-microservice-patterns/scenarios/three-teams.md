# E2: 三个团队的微服务决策故事

> ⚠️ 以下均为模拟场景，基于常见工程实践构造，非真实事件。

## A 团队：10 人创业公司，拆早了

**背景**：Go + PostgreSQL 的电商单体，日活 3 万，QPS 峰值 200。团队 10 人，全栈。

**决策过程**：CTO 参加技术大会，听了"微服务是大势所趋"的分享。回来后决定"趁项目还不大，早拆早受益"。用了两个月把单体拆成 8 个服务：用户、商品、订单、库存、支付、通知、搜索、推荐。

**结果**：
- 原来部署 1 个容器，现在部署 8 个 + 1 个 API Gateway + 1 个服务发现
- 原来一个人能 debug 全流程，现在追一个 bug 要翻 3 个服务的日志
- 原来数据库一个事务搞定，现在订单和库存需要分布式事务
- 半年后，回滚到单体，但 Kubernetes 集群已经买了一年

**用5问检查法复盘**：0/5 → 根本不应该拆

## B 团队：50 人电商平台，拆对了

**背景**：Go + MySQL 的电商平台，日活 200 万，QPS 峰值 5 万。3 个开发组：交易组（15人）、商品搜索组（20人）、推荐组（15人）。

**决策过程**：搜索组和交易组频繁出现代码合并冲突，推荐系统的每日迭代被交易系统的周发布拖累。架构师用拆分检查法评估后，决定拆 3 个服务（不是 8 个）。

**结果**：
- 搜索服务独立扩到 20 个实例，交易服务 5 个实例——资源利用率提升 40%
- 推荐服务独立发布后，迭代速度从每周 1 次提升到每天 2 次
- 三个组代码冲突降为零

**用5问检查法复盘**：5/5 → 每个维度都有明确收益

## C 团队：30 人 SaaS，选错通信方式

**背景**：B 团队拆分成功后，C 团队（同公司另一个产品线）照搬经验。拆了 5 个服务，但所有服务间通信都用 gRPC。

**决策过程**："gRPC 性能好，Google 在用，直接上。"没有区分同步/异步场景。

**问题**：
- 用户下单→发短信通知：gRPC 同步调用，短信网关响应慢（2-3s），直接拖累下单接口的 P99
- 订单状态变更→多个下游消费：gRPC 只能点对点调用，需要订单服务逐个通知 4 个下游，任何一个慢都影响响应
- 配置变更→全服务广播：gRPC 没有发布/订阅能力，改成轮询，浪费资源

**修复**：
- 通知类场景改用 NATS（异步，解耦）
- 订单状态变更改用 Kafka（发布订阅，可靠投递）
- gRPC 只保留在"订单→库存"这种同步强一致场景

**教训**：协议选型不是"哪个性能好用哪个"，是"哪个场景用哪个"。
