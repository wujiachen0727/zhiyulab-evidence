# 通信方式对比 Benchmark 结果

## 环境
[实测 Go 1.26.2, Apple M4 Pro, darwin/arm64]

## 原始数据（3 轮取均值）

| 方式 | 延迟 (ns/op) | 内存分配 (B/op) | 分配次数 (allocs/op) |
|------|:-----------:|:--------------:|:-------------------:|
| HTTP/JSON（httptest 本地回环） | ~65,388 | ~9,346 | 101 |
| 模拟 gRPC（函数调用+JSON序列化） | ~67.8 | 32 | 1 |
| 异步 Channel | ~204.8 | 0 | 0 |

## 解读

⚠️ **重要提醒**：这组 benchmark 对比的不是三种协议的绝对性能，而是**通信方式本身的开销差异**。

1. **HTTP/JSON 的开销主要来自网络栈**（即使是本地回环）：~65μs/请求，101 次内存分配。真实跨网络场景会更高。
2. **gRPC 模拟版**仅展示序列化开销（函数调用级），真实 gRPC 有网络栈但比 HTTP 轻（HTTP/2 + protobuf）。真实差距大约在 3-10x 而非 1000x。
3. **异步 Channel** 几乎零开销，但代价是"调用方不知道结果什么时候回来"。

## 核心结论

- 选 HTTP/JSON 还是 gRPC，**不是看绝对性能**（两者在大多数业务场景下都够用），而是看**序列化类型安全性和代码生成收益**
- 选同步还是异步，才是真正影响架构的决策——异步意味着"最终一致"，需要处理消息重复和顺序问题
- 真正的性能瓶颈几乎不在通信协议本身，而在**业务逻辑和数据库查询**
