# E4: 拆分决策5问检查法

## 推演过程

[推演] 从 DDD 限界上下文理论 + 工程实践约束推导

### 核心逻辑

拆分的本质是**把一个进程内的函数调用变成跨网络的服务调用**。这个变换有代价（网络延迟、错误处理、运维复杂度），所以需要足够的收益来覆盖成本。

### 5问检查法

每个候选拆分点，问自己这5个问题：

1. **数据独立吗？** — 这两块业务的数据是否可以分开存储，互不依赖？
   - Yes → +1（拆分后不需要分布式事务）
   - No → 0（拆了还得跨服务查数据，得不偿失）

2. **团队独立吗？** — 这两块业务是否由不同团队负责？
   - Yes → +1（减少代码合并冲突和沟通成本）
   - No → 0（同一个团队维护两个服务，运维成本翻倍）

3. **发布独立吗？** — 这两块的发布节奏是否不同？
   - Yes → +1（独立部署是微服务的核心收益）
   - No → 0（一起发布的话微服务只增加了部署复杂度）

4. **瓶颈隔离吗？** — 这两块有没有明显的性能差异需要独立扩缩容？
   - Yes → +1（比如商品搜索 QPS 是下单的 100 倍）
   - No → 0（单体也能水平扩展）

5. **故障域隔离吗？** — 一块挂了是否不应该影响另一块？
   - Yes → +1（比如推荐系统挂了不应该影响下单）
   - No → 0（本来就强依赖，拆了也隔离不了）

### 决策规则

| 得分 | 建议 |
|:----:|------|
| 0-1 | 不拆——收益不足以覆盖成本 |
| 2 | 谨慎——可以考虑，但要严格评估每个 No 的影响 |
| 3-5 | 拆——收益明确，风险可控 |

### 验证：用三个团队走一遍

**A 团队（10人电商，全栈）**：
- 数据独立？No（订单和库存共享数据库）
- 团队独立？No（就一个团队）
- 发布独立？No（一起发布）
- 瓶颈隔离？No（QPS 都差不多）
- 故障域隔离？No（订单和库存强耦合）
- **得分：0 → 不拆**

**B 团队（50人，订单组+库存组+推荐组分开）**：
- 数据独立？Yes（各自的数据库）
- 团队独立？Yes（三个组）
- 发布独立？Yes（推荐系统每天发布，订单每周发布）
- 瓶颈隔离？Yes（推荐 QPS 是订单的 50 倍）
- 故障域隔离？Yes（推荐挂了不影响下单）
- **得分：5 → 拆**
