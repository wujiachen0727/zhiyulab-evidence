# E6: 通信选型决策树

## 推演过程

[推演] 从通信模式的三个核心维度推导决策路径

### 三个维度

1. **同步 vs 异步** — 调用方是否需要立即拿到结果？
2. **强一致 vs 最终一致** — 数据一致性要求多高？
3. **延迟敏感度** — 能忍受多少延迟？

### 决策树

```
调用方需要立即拿到结果？
├── Yes（同步调用）
│   ├── 高性能要求（P99 < 10ms, 高 QPS）？
│   │   ├── Yes → gRPC（二进制序列化 + HTTP/2 多路复用）
│   │   └── No → HTTP/JSON（简单、通用、调试友好）
│   └── 需要流式数据？
│       ├── Yes → gRPC Streaming
│       └── No → gRPC Unary / HTTP
└── No（异步调用）
    ├── 需要保证消息不丢？
    │   ├── Yes（强可靠）
    │   │   ├── 需要消费者组/顺序消费？
    │   │   │   ├── Yes → Kafka（分区保序 + 消费者组）
    │   │   │   └── No → RabbitMQ / NATS JetStream
    │   │   └── 消息量极大（> 10w/s）？
    │   │       ├── Yes → Kafka
    │   │       └── No → RabbitMQ / NATS JetStream
    │   └── No（可丢弃）→ NATS Core（最轻量）
    └── 需要发布/订阅模式？
        ├── 多消费者 → Kafka / NATS JetStream
        └── 单消费者 → 任意 MQ
```

### 速查表

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 订单→库存（扣减） | gRPC | 同步、强一致、低延迟 |
| 订单→通知（短信/邮件） | NATS/Kafka | 异步、最终一致、允许延迟 |
| 用户→推荐（列表） | HTTP/gRPC | 同步、非强一致、高QPS |
| 日志采集→分析 | Kafka | 异步、大量、允许延迟 |
| 配置变更→广播 | NATS Pub/Sub | 发布订阅、轻量 |

### 常见错误

1. **gRPC 一把梭**：所有场景都用 gRPC → 异步通知场景变成同步阻塞，P99 飙升
2. **Kafka 一把梭**：所有场景都用 Kafka → 简单查询也走消息队列，响应时间从 5ms 变 500ms
3. **HTTP 一把梭**：所有场景都用 HTTP/JSON → 高 QPS 场景序列化开销明显
