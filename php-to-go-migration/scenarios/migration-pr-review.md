# 场景模拟：一次 PHP 到 Go 的迁移 PR 评审

> 类型：场景模拟，基于常见后端迁移场景构造。无真实公司、人名和业务数据。

## 场景背景

一个用户资料更新接口从 PHP 服务迁到 Go。接口本身不复杂：接收 `user_id`、`email`、`nickname`，更新用户资料并记录审计日志。

迁移前，PHP 版本的大部分边界靠框架、运行时和团队经验兜底：

- 请求体先进入框架的 request 对象。
- 字段从数组里取出来，必要时在业务函数里转类型。
- 校验失败由框架异常或统一错误处理中间件兜底。
- 只要 happy path 能跑，很多边界问题会留到联调或线上日志里暴露。

迁移后，Go 版本在 PR 阶段就被迫回答更多问题：

- 请求结构体字段类型是什么？
- JSON 解码失败如何返回？
- `user_id` 是路径参数还是请求体？
- `context.Context` 如何传到 repository？
- 更新失败和审计日志失败谁影响主流程？
- 错误是否能被上层识别，而不是只剩字符串？

## PR 评审对话骨架

### 第一轮：字段边界

Reviewer 看到 Go 代码里的请求结构体：

```go
type UpdateProfileRequest struct {
    UserID   int64  `json:"user_id"`
    Email    string `json:"email"`
    Nickname string `json:"nickname"`
}
```

他没有先问“能不能跑”，而是问：

> 如果前端把 `user_id` 传成字符串，这个请求应该在 JSON 解码阶段失败，还是兼容转换？

这个问题在 PHP 版本里可能不会第一时间出现。因为数组、字符串和数字之间的转换更容易被延后到业务函数里。但 Go 的结构体把边界提前摆出来：你必须决定输入契约，而不是先让数据流进去。

### 第二轮：失败路径

迁移前，业务代码可能更关注主路径：

```php
$user = $repo->find($request['user_id']);
$user->email = $request['email'];
$repo->save($user);
```

迁移后，Go PR 里每一步失败都必须被处理：

```go
user, err := repo.Find(ctx, req.UserID)
if err != nil {
    return err
}

if err := repo.Save(ctx, user); err != nil {
    return err
}
```

Reviewer 追问的不是“为什么 Go 这么啰嗦”，而是：

> `Find` 的 not found、DB timeout、权限不匹配，是不是应该被上层区分？

这里的复杂度没有消失。它从“统一异常兜底”移动到了错误类型、错误包装和上层处理策略里。

### 第三轮：副作用边界

更新资料后需要写审计日志。PHP 版本里这一步可能藏在框架事件或 service 里。Go PR 里 reviewer 会逼着作者说明：

> 审计日志失败，要不要让用户资料更新失败？如果不失败，怎么补偿？如果失败，用户看到什么错误？

这就是复杂度归属变化最明显的地方。Go 没有替你决定副作用边界。它只是让你更难假装这个边界不存在。

## 这个场景支撑的子论点

1. 从 PHP 到 Go，很多问题不是变多了，而是更早被看见了。
2. Go 的显式边界会把“以后再说”的问题提前放到 PR 里。
3. 迁移的认知变化不是“我会写 Go 语法了”，而是“我开始更早定义输入、失败和副作用的责任边界”。

## 正文使用建议

不要把这个场景写成“PHP 代码差，Go 代码好”。更稳的写法是：

> PHP 版本不是没有边界，而是很多边界被框架和团队经验吸收了。Go 版本不是天然更高级，而是它让这些边界更难隐身。
