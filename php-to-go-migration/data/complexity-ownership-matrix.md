# 复杂度归属矩阵

> 类型：逻辑推演 + 写作框架。用于正文中解释“复杂度没有消失，只是换了位置”。

| 复杂度类型 | PHP 常见默认归属 | Go 常见默认归属 | 迁移后的认知变化 |
|------------|------------------|----------------|------------------|
| 输入类型错误 | 请求数组、运行时转换、框架校验、业务函数兜底 | JSON 解码、struct 字段类型、显式校验 | 先定义输入契约，再写业务逻辑 |
| 错误处理 | 异常、中间件、框架统一错误处理 | 函数返回 `error`，调用点逐层决定 | 失败路径从“兜底机制”变成主流程的一部分 |
| 空值和缺字段 | 运行时 notice/warning、默认值、框架 helper | 零值、指针、可选字段、校验函数 | 必须区分“没传”和“传了零值” |
| 并发共享状态 | 传统 PHP 请求模型下较少显式暴露 | goroutine、channel、mutex、context | 共享状态边界必须提前设计 |
| 副作用失败 | 事件、队列、框架插件或统一异常 | 显式返回、补偿策略、上下文取消 | 主流程和副作用的失败关系必须说清楚 |
| 团队风格 | 依赖框架约定、代码规范、review 习惯 | gofmt、包边界、接口、显式错误 | 风格争议减少，边界设计争议增加 |

## 推演结论

Go 没有让复杂度凭空减少。它更常做的事，是把复杂度从运行时、框架兜底和团队经验里拉出来，放到类型定义、错误处理、测试、代码审查和并发设计里。

这也是“从 PHP 到 Go”的核心认知变化：你不再只问“这段代码能不能跑”，而是更早追问“这段代码的责任边界在哪里”。
