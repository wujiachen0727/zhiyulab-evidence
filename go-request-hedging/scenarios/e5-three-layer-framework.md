# E5 三层决策框架推导（逻辑推演）

## 推演前提

1. P99 尾延迟的成因可以按"距离根因的远近"分类
2. 不同层级的成因需要不同层级的治理工具
3. 跳层治理 = 症状治疗，只能暂时掩盖不能根治

## 推导链条

### 第一步：尾延迟的成因分类

P99 高，本质上是"个别请求慢"。为什么个别请求会慢？三类原因：

**病因层（根因）**：请求处理过程中本身有问题
- GC 停顿（STW）
- 锁竞争（mutex 等待）
- 慢查询（数据库索引缺失、全表扫描）
- 资源耗尽（CPU 满载、内存不足）

**放大层（结构）**：系统结构把"个别慢"放大成"整体慢"
- Fan-out 放大：1% × 100 节点 = 63% 整体超时（E3 实测验证）
- 缓存击穿：热点 key 失效瞬间所有请求穿透到下游
- 批处理等待：批量接口因单个慢元素拖慢整体

**症状层（表现）**：客户端感知到的"慢"
- P99 数字本身
- 超时率
- 用户感知的卡顿

### 第二步：每层有独立的治理工具

**病因层工具**：直接修复根因
- GC 调优（GOGC、GOMEMLIMIT）
- 锁优化（分片锁、无锁结构、降低临界区粒度）—— E2 实测验证
- 慢查询优化（索引、SQL 重写、读写分离）
- 资源扩容（垂直/水平）

**放大层工具**：阻断"个别慢"到"整体慢"的传播
- Fan-out 拆分（降低并行度，或改串行为并行）
- 缓存预热（避免冷启动击穿）
- 批处理改造（流式返回、超时降级）
- 超时控制（单节点超时切断，避免拖慢整体）

**症状层工具**：在客户端"绕过"慢请求
- hedging（发第二个请求，取先到的）
- fallback（降级到缓存或默认值）
- circuit breaker（熔断，停止发送）

### 第三步：跳层治理 = 症状治疗

如果 P99 高是因为锁竞争（病因层），直接上 hedging（症状层）会发生什么？

- hedging 降了客户端 P99（症状缓解）✓
- 但锁竞争还在（病因未除）✗
- 更糟的是：hedging 带来近一倍额外请求，全部灌进同一把锁（病因恶化）✗
- E1 实测验证：mutex 等待时间增加 1147%

**类比**：发烧吃退烧药。体温计数字好看了（症状缓解），但感染还在（病因未除），而且退烧药压制免疫反应可能让感染恶化（跳层治理的副作用）。

### 第四步：正确的治理顺序

三层递进，不能跳层：

1. **先治病因**：定位 P99 高的根因（profiling、慢查询日志、GC trace），直接修复
2. **再消放大**：检查系统结构是否有放大效应（Fan-out、缓存击穿），用结构改造阻断传播
3. **最后兜底症状**：病因和放大都治理好后，hedging 作为"保险"兜底真正的长尾

### 第五步：hedging 的正确角色

hedging 不是"第一选择"，是"最后兜底"。

- 病因层未修复时：hedging 触发率 > 50%，代价是资源翻倍 + 病因恶化
- 病因层修复后：hedging 触发率 5-15%，回归"保险"角色，只在真正长尾时触发
- E2 实测验证：修复锁竞争后，hedging 触发率从 99.6% 降到 14.7%

## 框架总结

```
尾延迟治理三层框架
├── 病因层（根因）—— GC/锁/慢查询/资源
│   └── 工具：profiling + 直接修复
├── 放大层（结构）—— Fan-out/缓存击穿/批处理
│   └── 工具：结构改造 + 超时控制
└── 症状层（表现）—— P99 数字
    └── 工具：hedging + fallback + 熔断
```

**判断规则**：如果 hedging 触发率 > 50%，说明你在用症状层工具治病因层的病——方向错了。

## 标注

- [逻辑推演] 本框架是原创认知模型，基于工程实践归纳
- 推理链 ≤ 3 步：成因分类 → 工具匹配 → 跳层后果
- 实验支撑：E1 证伪实验（跳层治理的副作用）、E2 三层对比（修复后 hedging 角色变化）、E3 Fan-out 放大（放大层的数学验证）
