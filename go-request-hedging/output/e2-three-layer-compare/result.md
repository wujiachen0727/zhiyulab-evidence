# E2 三层对比实验结果

**实验时间**：2026-06-28 16:44
**环境**：Go 1.26.4 darwin/arm64
**实验代码**：`evidence/code/e2-three-layer-compare/`

## 实验设计

三种方案对比：
- **A 无优化基线**：全局单锁 + 无 hedging
- **B 仅 hedging**（症状治疗）：全局单锁 + hedging
- **C 修复锁竞争 + hedging**（病因治疗 + 症状兜底）：16 分片锁 + hedging

每方案 2000 请求，并发 50，服务端长尾分布同 E1（90% 概率 10-50ms，10% 概率 200-500ms）。

## 原始数据

### 场景 A：无优化基线

```
P50=55.16ms
P95=364.89ms
P99=482.90ms
n=2000, 服务端处理 2000, hedging 触发 0
```

### 场景 B：仅 hedging（症状治疗）

```
P50=155.96ms
P95=206.31ms
P99=387.20ms
n=2000, 服务端处理 3992, hedging 触发 1993
```

### 场景 C：修复锁竞争 + hedging（病因治疗 + 症状兜底）

```
P50=34.36ms
P95=82.33ms
P99=103.59ms
n=2000, 服务端处理 2265, hedging 触发 294
```

## 三层对比

| 方案 | P50 | P95 | P99 | 服务端请求量 | hedging 触发 |
|------|:---:|:---:|:---:|:---:|:---:|
| A 无优化 | 55.2ms | 364.9ms | 482.9ms | 2000 | 0 |
| B 仅 hedging | 156.0ms | 206.3ms | 387.2ms | 3992 | 1993 |
| C 修复+hedging | 34.4ms | 82.3ms | 103.6ms | 2265 | 294 |

## 相对降幅

| 对比 | P99 降幅 | P50 降幅 | 服务端额外负载 |
|------|:---:|:---:|:---:|
| B vs A（仅 hedging） | -19.8% | **+182%**（变差） | +99.6%（翻倍） |
| C vs A（修复+hedging） | **-78.5%** | -37.6% | +13.3% |
| C vs B（修复 vs 不修复） | -73.3% | -78.0% | -43.3% |

## 结论

1. **仅 hedging（方案 B）效果有限且有代价**：P99 只降 20%，但 P50 变差 3 倍，服务端负载翻倍
2. **修复病因 + hedging（方案 C）效果决定性**：P99 降 78.5%，P50 也降 37.6%，服务端负载只增 13%
3. **hedging 的角色变化**：方案 B 中 hedging 触发 1993 次（99.6%，几乎是主力）；方案 C 中 hedging 触发 294 次（14.7%，回归兜底角色）
4. **核心洞察**：hedging 本身是兜底工具，当病因层未修复时被迫成为主力，代价是 P50 变差和服务端负载翻倍；病因层修复后，hedging 回归"保险"角色，只在真正长尾时触发

## 标注

- [实测 Go 1.26.4 darwin/arm64]
- 分片锁用 16 个分片（shardCount=16），key 用递增计数器取模
- 本实验用 time.Sleep 模拟慢操作，真实生产中修复病因的手段包括：分片锁、无锁结构、降低临界区粒度、GC 调优、慢查询索引等
