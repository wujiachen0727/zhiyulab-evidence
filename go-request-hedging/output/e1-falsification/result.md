# E1 证伪实验结果

**实验时间**：2026-06-28 16:42
**环境**：Go 1.26.4 darwin/arm64，macOS 本地开发机
**实验代码**：`evidence/code/e1-falsification/`

## 实验设计

- 服务端：HTTP handler 有一个全局 mutex 保护短临界区（1-3ms），临界区外有慢操作（90% 概率 10-50ms，10% 概率 200-500ms 长尾）
- 场景 A：客户端无 hedging，发 2000 请求，并发 50
- 场景 B：客户端有 hedging（hedgeDelay=50ms），发 2000 请求，并发 50
- 采集：客户端 P50/P95/P99，服务端 mutex profile（runtime.SetMutexProfileFraction(100)）

## 原始数据

### 场景 A：无 hedging

```
P50=50.382208ms
P95=375.770167ms
P99=486.429584ms
n=2000, 服务端处理 2000
mutex profile 总等待时间: 37.29s
```

### 场景 B：有 hedging

```
P50=156.010583ms
P95=207.123042ms
P99=237.517666ms
n=2000, 服务端处理 3991, hedging 触发 1991 次
mutex profile 总等待时间: 465.15s
```

## 对比

| 指标 | 场景 A���无 hedging） | 场景 B（有 hedging） | 变化 |
|------|:---:|:---:|:---:|
| 客户端 P50 | 50.4ms | 156.0ms | ↑ 210%（P50 变差，因 hedging 增加排队） |
| 客户端 P95 | 375.8ms | 207.1ms | ↓ 44.9% |
| 客户端 P99 | 486.4ms | 237.5ms | **↓ 51.2%** |
| 服务端处理请求量 | 2000 | 3991 | ↑ 99.5%（hedging 导致近一倍额外请求） |
| 服务端 mutex 总等待时间 | 37.29s | 465.15s | **↑ 1147%** |

## 结论

**假设被强力支撑**：hedging 是症状治疗不是病因治疗。

1. **P99 下降 51.2%**：hedging 确实降低了客户端感知的 P99 延迟
2. **mutex 等待时间暴增 1147%**：病因（锁竞争）不仅没解决，反而因为 hedging 带来的近一倍额外请求量，锁竞争恶化了 12 倍
3. **P50 反而变差 210%**：hedging 让普通请求也承担了额外排队成本
4. **服务端处理量翻倍**：hedging 触发 1991 次（接近 100% 触发率），意味着大量额外负载

**关键洞察**：hedging 在客户端"看起来"降了 P99，但在服务端"实际上"加剧了病因。如果只看客户端 P99 监控，你会以为问题解决了；但服务端的锁竞争 profile 会告诉你真相——病因不仅还在，而且更严重了。

## 标注

- [实测 Go 1.26.4 darwin/arm64]
- 本实验用 time.Sleep 模拟慢操作，真实生产环境中慢操作可能来自下游 RPC、数据库查询、GC 停顿等
- mutex 采样率 1/100，实际等待时间是 profile 值的约 100 倍，但两个场景的对比比例有效
- P50 变差是 hedging 的已知副作用（额外请求增加排队），本文会在"hedging 成本"章节展开
