# E6 hedging 成本实测结果

**实验时间**：2026-06-28 17:01
**环境**：Go 1.26.4 darwin/arm64
**实验代码**：`evidence/code/e6-hedging-cost/`

## 实验设计

对比有无 hedging 的资源成本：
- 客户端连接池限制：MaxIdleConns=100, MaxIdleConnsPerHost=50, MaxConnsPerHost=100
- 2000 请求，并发 50
- 采集：延迟分位数、内存分配、GC 次数、hedging 触发次数

## 原始数据

| 指标 | 无 hedging | 有 hedging | 变化 |
|------|:---:|:---:|:---:|
| P50 | 57.5ms | 156.2ms | ↑ 172%（变差） |
| P95 | 356.7ms | 207.0ms | ↓ 42% |
| P99 | 478.3ms | 236.6ms | ↓ 50.5% |
| hedging 触发 | 0 | 1987 | — |
| hedging 触发率 | 0% | 99.4% | — |
| 服务端处理量（推算） | 2000 | ~3987 | ↑ 99.4% |
| 内存分配 | 13.99 MB | 77.67 MB | **↑ 455%** |
| GC 次数 | 8 | 18 | **↑ 125%** |

## 成本分类

### 1. 连接池压力

- 额外请求量：~1987（99.4% 触发率）
- 连接池需要承受近一倍的额外连接
- 下游服务 QPS 被放大近一倍
- 真实生产中：连接池打满 → 请求排队 → 新的长尾 → 触发更多 hedging → 恶性循环

### 2. 内存与 GC 压力

- 内存分配增加 455%（13.99MB → 77.67MB）
- GC 次数增加 125%（8 次 → 18 次）
- 真实生产中：GC 压力上升 → STW 停顿增加 → 新的长尾源（正是 hedging 想治的病）

### 3. 可观测性污染

- hedging 触发率 99.4% 意味着服务端日志量近翻倍
- 服务端 metrics 失真：处理量虚高、P99 失真（服务端 P99 被 hedging 请求拉高）
- 链路追踪出现重复 span：同一逻辑请求对应两条服务端 trace，排障时容易误判
- 真实生产中：告警阈值需要重调，SLO 计算需要区分原始请求和 hedging 请求

## 结论

hedging 不是免费的：

1. **P50 变差 2.7 倍**：为了降 P99，牺牲了普通请求的体验
2. **内存分配增加 4.5 倍**：每个 hedging 请求都是完整的 HTTP 请求，内存成本翻倍
3. **GC 压力增加 1.25 倍**：更多内存分配 → 更频繁 GC → 新的长尾源
4. **可观测性全面污染**：日志、metrics、trace 都需要额外处理

**核心洞察**：hedging 的成本不是"5% 额外请求"那么轻巧（这是《The Tail at Scale》论文的乐观估计）。在高触发率场景下，hedging 的成本是接近翻倍的资源消耗 + 可观测性系统的全面污染。

## 标注

- [实测 Go 1.26.4 darwin/arm64]
- 论文《The Tail at Scale》提到的"5% 额外请求"是基于 Google 生产环境的低触发率场景，本实验的 99.4% 触发率是"病因未修复"时的极端情况
- 真实生产中，合理的 hedging 触发率应在 5-15%（E2 方案 C 实测 14.7%）
