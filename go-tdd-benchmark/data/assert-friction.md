# E3: assert 摩擦成本实测

[实测 Go 1.26.2 + testify v1.11.1] 2026-04-14

## 测试方法

用 5 个常见断言场景（整数比较、字符串比较、浮点数比较+错误检查、错误断言、切片比较），分别用纯 testing 包和 testify 编写，统计代码量差异。

环境：Go 1.26.2 darwin/arm64, testify v1.11.1

## 结果

| 维度 | 纯 testing 包 | testify |
|------|:------------:|:-------:|
| 代码行数 | 51 行 | 32 行 |
| 断言行数占比 | ~80% | ~60% |
| 需要手写的样板代码 | if + Errorf 格式串 | 无 |
| 错误信息质量 | 自定义（手写格式串） | 自动生成（含期望值和实际值） |

**代码量差异**：testify 写法比纯 testing 包少约 37% 代码。

## 结论

1. **摩擦真实存在**：同一组断言，纯 testing 包需要多写约 37% 的代码，主要是 if 判断 + Errorf 格式化。
2. **但摩擦不大**：37% 不是"翻倍"级别的差距，且 Go 开发者对 if+Errorf 模式已很熟悉。
3. **收益权衡**：testify 省代码但引入外部依赖，纯 testing 包多写几行但零依赖。Go 标准库选择了零依赖路线（E2 已验证：0 处 testify 使用）。
4. **t.Helper() 是 Go 的答案**：标准库 2559 处 t.Helper() 说明 Go 不是"不需要断言"，而是"自己封装断言辅助函数，框架不替你做"。
