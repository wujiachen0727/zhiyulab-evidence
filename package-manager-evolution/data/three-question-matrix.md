# 三问矩阵：包管理器到底在担保什么

> 类型：逻辑推演 / 独立论据  
> 标注：矩阵中的事实项来自本地生成文件与官方文档；解释项为工程推演。

## 核心三问

| 问题 | 如果答不上来，工程风险是什么 | 对应能力层 |
|------|----------------------------|-----------|
| 版本谁决定？ | 同一份声明在不同时间解析出不同依赖，CI 和线上环境漂移 | 快照层 / 协议层 |
| 来源谁证明？ | 包内容被替换、镜像污染或下载来源不可追溯 | 协议层 |
| 构建如何复现？ | 昨天能过的构建今天失败，团队无法判断变化来自业务代码还是依赖树 | 快照层 / 协议层 |

## 四生态对照

| 生态 | 版本谁决定 | 来源谁证明 | 构建如何复现 | 文章里的定位 |
|------|------------|------------|--------------|--------------|
| npm | package.json 给范围，package-lock.json 固化实际解析树 | package-lock.json 中的 resolved / integrity 字段 | 提交 package-lock.json，CI 使用同一锁文件安装 | 典型“快照层”：把 node_modules 解析结果变成可审查输入 |
| Python/pip + pylock | requirements/pyproject 表达需求，pip lock/pylock.toml 固化当前环境解析结果 | pylock.toml 记录 wheel URL 与 sha256 | pylock.toml 面向可复现安装，但 pip lock 仍标注实验性且受平台/Python 版本约束 | 从安装器心智向标准锁文件迁移的生态样本 |
| Go Modules | go.mod 记录最小版本要求，MVS 规则决定 build list | go.sum 记录模块校验和 | 相同 go.mod + MVS + go.sum 校验形成确定性路径 | 关键反例：没有传统依赖树 lockfile，也能形成确定性协议 |
| Cargo | Cargo.toml 表达版本范围，Cargo.lock 固化具体解析版本 | Cargo.lock 记录 registry source 与 checksum | 提交 Cargo.lock，构建使用锁定依赖图 | 构建系统心智强：声明范围和构建快照职责清晰分离 |

## 推演结论

包管理器越成熟，越不像“下载器”。

它会把“我想要什么依赖”“这次实际拿到了什么”“这些内容能不能被证明没有变过”拆成不同文件和规则。拆得越清楚，团队越能把依赖变化纳入代码审查。

这也是本文的核心判断：现代包管理器的演进，不是命令语法的演进，而是构建信任边界的演进。
