# 工程规范经验落地：依赖升级 PR 怎么审

> 类型：经验落地 / 独立论据  
> 标注：这是工程规范视角的实践总结，不编造具体公司、人名或历史事件。

## 核心判断

依赖升级 PR 不应该只看业务代码。

如果 lockfile、go.sum、Cargo.lock、pylock.toml 发生变化，它们就是本次变更的一部分。忽略这些文件，本质上是在忽略构建输入。

## 审查顺序

| 步骤 | 看什么 | 要回答的问题 |
|------|--------|-------------|
| 1 | 直接依赖声明 | 这次是主动升级，还是间接漂移？ |
| 2 | 锁定/校验文件 diff | 传递依赖变了多少？是否出现新来源或新 hash？ |
| 3 | CI 安装命令 | CI 是否使用锁文件/校验文件，而不是重新自由解析？ |
| 4 | 回滚路径 | 如果依赖变更导致问题，能否回到上一份构建输入？ |
| 5 | 触发条件 | 这次升级是修漏洞、修 bug，还是无理由刷新？ |

## 三条可写进正文的规则

1. **lock diff 不是噪音**：它是依赖树变化的审计记录。
2. **依赖升级要有理由**：安全修复、兼容性修复、功能需要都可以；“顺手升级”不应该进入主干。
3. **CI 必须使用确定性入口**：npm 用锁文件安装，Cargo 提交 Cargo.lock，Go 保留 go.sum，Python 明确 pylock/requirements 的边界。

## 和本文论点的关系

这部分支撑文章结尾：包管理器不是下载器，而是团队构建信任的协议。协议的最终落点不是文件本身，而是团队如何把文件变化纳入工程流程。
