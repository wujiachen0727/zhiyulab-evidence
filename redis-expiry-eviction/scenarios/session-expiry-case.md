# E5 经验落地：Session Key 集中过期导致缓存雪崩

## 场景背景
线上 Redis 缓存了大量用户 session（约 20 万个 key，每个约 500 字节），统一设置了 8 小时 TTL。

## 问题表现
某天下午 3 点，接口 P99 延迟从 20ms 飙升到 2s+，CPU 使用率从 30% 跳到 95%。大量请求打到数据库。

## 排查过程
1. 发现 Redis expired_keys 指标在短时间内飙高
2. 检查 key 的 TTL 分布——20 万个 session key 集中在同一时间段过期
3. 定期删除线程忙不过来，大量过期 key 未被及时清理
4. 内存未释放，淘汰器触发后开始淘汰活跃 key
5. 大量缓存未命中 → 请求穿透到数据库 → 数据库连接池打满

## 根因分析
- 直接原因：20 万个 key 设置了相同的 TTL（8 小时），导致同时过期
- 深层原因：惰性删除来不及处理这么多同时过期的 key（除非全部被访问）
- 连锁反应：定期删除的 CPU 时间限制 25% 保护 → 过期 key 清理滞后 → 内存未释放 → 淘汰器触发

## 解决方案
1. 给 session key 的 TTL 加入随机偏移（±30%）
2. 对 key 做分片（hash 到不同前缀），避免集中扫描
3. 设置 maxmemory-policy 为 allkeys-lru（至少不会因为淘汰策略选错导致业务受影响）

## 教训
"过期时间到了 key 就被删"是一个危险的错觉。过期策略的惰性+定期删除机制意味着：**过期时间只是"保证 key 在这个时间后不可读"，不保证"到这个时间就释放内存"**。
