# 场景模拟：3 种业务场景的插件化决策

> 标注：这是模拟场景，基于合理的业务逻辑构造

## 场景 1：配置热更新

**背景**：一个 API 网关需要动态调整限流阈值、路由规则，不能重启服务。

**常见诉求**："我需要插件来实现配置热更新"

**决策推理**：

1. 需求本质：运行时改变行为，无需重启
2. 配置变更是数据变更，不是逻辑变更
3. 数据变更的解决方案：配置中心（etcd/consul/nacos）+ 内存热加载
4. 配置中心的延迟：etcd watch 通知 < 10ms，远优于任何插件方案的加载时间
5. 无需加载代码 → 不需要插件

**结论**：配置热更新不需要插件系统，配置中心是更优解。
**方案推荐**：配置中心 + 内存热加载

## 场景 2：规则引擎

**背景**：一个风控系统需要动态添加审核规则，规则逻辑可能很复杂（条件分支、函数调用）。

**常见诉求**："规则经常变，需要插件化来动态加载规则"

**决策推理**：

1. 需求本质：运行时添加业务规则
2. 规则是"受限逻辑"——不是任意代码，是结构化条件
3. 受限逻辑的解决方案：DSL/表达式引擎
4. Go 生态的选项：
   - govaluate：表达式引擎，~50ns/op
   - cel-go：Google CEL，~200ns/op，功能更强
   - 自定义 DSL：按需，最灵活
5. 表达式引擎 vs 插件：
   - 表达式引擎：安全沙箱、可序列化、可版本管理、可热替换
   - 插件：任意代码执行、难以审计、不可卸载

**结论**：规则引擎场景不需要插件系统，DSL/表达式引擎在安全性和可维护性上全面优于插件。
**方案推荐**：CEL/govaluate + 规则存储（数据库/配置中心）

## 场景 3：第三方扩展

**背景**：一个 DevOps 平台需要支持第三方开发者编写自定义集成（如对接内部监控系统、自定义部署流程）。

**常见诉求**："需要开放 API 让第三方写插件"

**决策推理**：

1. 需求本质：让不可信的第三方代码在平台内运行
2. 关键约束：
   - 第三方代码不可信 → 需要沙箱隔离
   - 第三方代码质量不可控 → 不能让它崩溃影响主进程
   - 第三方代码需要独立部署 → 版本独立
3. 方案对比：
   - 原生 plugin：无沙箱，崩溃会拖垮主进程 → ❌ 不可接受
   - go-plugin (RPC)：进程隔离 + 崩溃隔离 → ✅ 安全
   - yaegi：解释器模式，有一定隔离 → ⚠️ 隔离不够彻底
   - wazero (WASM)：沙箱隔离 + 合理性能 → ✅ 新兴方向
4. 延迟容忍度：第三方扩展通常是低频调用（每秒几次到几十次），400μs/call 完全可接受

**结论**：第三方扩展是真正需要插件化的场景。go-plugin (RPC) 是当前最成熟选择，wazero 是未来方向。
**方案推荐**：go-plugin (RPC) 作为主力，wazero 作为实验性方案

---

## 决策表

| 场景 | 需要插件? | 推荐方案 | 原因 |
|------|:---------:|---------|------|
| 配置热更新 | ❌ | 配置中心 | 数据变更 ≠ 逻辑变更 |
| 规则引擎 | ❌ | DSL/表达式引擎 | 受限逻辑不需要任意代码执行 |
| 第三方扩展 | ✅ | go-plugin (RPC) | 需要隔离 + 版本独立 |

**扩展到 5 维度**：

| 维度 | 配置热更新 | 规则引擎 | 第三方扩展 |
|------|----------|---------|----------|
| 调用频率 | 极高（每次请求） | 高（每条数据） | 低（每秒几次） |
| 扩展复杂度 | 低（改数值） | 中（加条件） | 高（任意逻辑） |
| 平台要求 | 全平台 | 全平台 | 可接受仅 Linux |
| 安全隔离 | 不需要 | 需要（防注入） | 必须（不可信代码） |
| 团队技能 | 低（配置文件） | 中（学习 DSL） | 高（Go 开发） |
