# 数据实测：RPC 开销在不同调用频率下的体感影响

> 标注：[推演] 基于实测 benchmark 数据 + 合理估算

## 基准数据

来源：go-plugin-benchmark + 本文实测

| 方案 | 延迟 (ns/op) | 延迟 (μs/op) |
|------|-------------:|-------------:|
| 原生调用 | ~1 | ~0.001 |
| plugin 接口 | ~1 | ~0.001 |
| yaegi | ~370 | ~0.37 |
| wazero | ~30 | ~0.03 |
| 管道 IPC（模拟） | ~2313 | ~2.3 |
| go-plugin (gRPC, 引用数据) | ~30000-50000 | ~30-50 |

## 不同调用频率下的影响

| 调用频率 | 原生 | yaegi | wazero | go-plugin (RPC) |
|---------|------|-------|--------|-----------------|
| 1 次/秒 | 1ns | 370ns | 30ns | 30-50μs |
| 100 次/秒 | 0.1μs | 37μs | 3μs | 3-5ms |
| 1000 次/秒 | 1μs | 370μs | 30μs | 30-50ms |
| 10000 次/秒 | 10μs | 3.7ms | 0.3ms | 300-500ms |

## 体感分析

### 1 次/秒（低频：第三方扩展、手动触发）
- go-plugin (RPC)：30-50μs → **完全无感**，延迟可忽略
- 结论：低频场景下 RPC 方案的开销没有任何体感影响

### 100 次/秒（中频：规则引擎、审批流）
- go-plugin (RPC)：3-5ms → **可感知但不影响体验**
- yaegi：37μs → **无感**
- 结论：中频场景 yaegi 性能可接受，RPC 开始有累积

### 1000 次/秒（高频：请求级拦截、数据处理）
- go-plugin (RPC)：30-50ms → **明显影响**，每秒浪费 3-5% 在通信上
- wazero：30μs → **无感**
- yaegi：370μs → **可接受**
- 结论：高频场景必须用原生/wazero/yaegi，RPC 方案不可行

### 10000 次/秒（极高：每请求多次调用）
- go-plugin (RPC)：300-500ms → **不可接受**，每秒 30-50% 浪费在通信
- yaegi：3.7ms → **开始有影响**
- wazero：0.3ms → **可接受**
- 结论：极高频率只有原生和 wazero 可行

## 关键结论

**RPC 开销的体感取决于调用频率，不取决于绝对延迟。**

- 低频（< 10次/秒）：所有方案都可接受，选最安全的
- 中频（10-1000次/秒）：yaegi/wazero 可接受，RPC 需评估
- 高频（> 1000次/秒）：只有原生/wazero 可行，RPC 不可行

这是决策表"调用频率"维度的核心依据。
