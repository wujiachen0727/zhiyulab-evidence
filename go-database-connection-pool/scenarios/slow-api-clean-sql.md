# 场景论据：慢查询日志干净，但接口 P99 飙升

## 场景用途

用于文章开头和第三章，引出“慢的可能不是 SQL，而是等 SQL 的队伍”。

## 场景脚本

一次接口延迟排查里，最容易误判的是这个场景：

- 接口 P99 从几十毫秒涨到几秒。
- 数据库慢查询日志里没有新 SQL。
- 单独拿 SQL 去数据库里跑，耗时也稳定。
- 应用日志只看到“数据库调用耗时高”，但没有区分“拿连接等待”和“执行 SQL”。

这时如果只盯着 SQL，你会继续查索引、查执行计划、查 Buffer Pool。

但真正该先看的可能是 `DB.Stats()`：

- `WaitCount` 是否持续增长？
- `WaitDuration` 是否占掉了大部分接口耗时？
- `OpenConnections` 是否已经贴近 `MaxOpenConns`？

如果这些信号同时出现，说明请求不是卡在数据库内部，而是卡在进入数据库之前：Go 连接池门口已经排队。

## 正文表达建议

不要写成“真实生产事故”。这是基于工程常识构造的典型排障场景，可用第一人称自然叙述：

> 我见过最容易误判的一类数据库慢，是慢查询日志很干净，接口 P99 却一路往上飙。你单独把 SQL 拿出来跑，它确实不慢。慢的不是 SQL，是请求在 Go 进程里等一个能用的连接。
