# 连接池等待实验数据

## 实验标注

[实测 Go 1.26.2 darwin/arm64]

查询耗时由自定义 `database/sql/driver` 固定 `time.Sleep(50ms)` 模拟。这个实验隔离观察 `database/sql` 连接池等待行为，不代表任何真实数据库的查询性能。

## 可用于正文的关键结论

1. **SQL 没变，P99 可以差两个数量级**：单次查询固定 50ms，`MaxOpen=1` 时 P99 达 4.60s；`MaxOpen=64` 时 P99 回到 53.3ms。
2. **等待指标能解释延迟差异**：`MaxOpen=1` 的累计 `WaitDuration` 为 227.70s，平均每次等待 1.43s；`MaxOpen=64` 的 `WaitCount/WaitDuration` 均为 0。
3. **调大连接数不是“优化口诀”，而是减少排队**：从 1 到 16 的改善，本质是队伍变短；当 `MaxOpen` 超过并发需求后，`WaitDuration` 归零，继续加连接不再改善单次查询耗时。
4. **`WaitCount` 不等于错误数**：本次所有场景 `errors=0`，但小连接池下 `WaitCount` 很高。连接池等待不会报错，却会进入应用延迟。

## 正文使用边界

- 可以写“我跑了一组模拟测试”，不能写成真实 MySQL/PostgreSQL 生产压测。
- 可以用来说明 `database/sql` 连接池等待如何进入应用延迟。
- 不能用来推荐生产环境 `MaxOpenConns=64`；生产参数还要看数据库连接上限、查询耗时、事务持有时间和代理层行为。
