# E1：gosec 扫描 10 个主流 Go 开源项目分布数据

## 实验方法

**扫描对象**：10 个 Top Star 的 Go 开源项目（截至 2026-04-30 主分支 HEAD）
- gin-gonic/gin（Web 框架）
- labstack/echo（Web 框架）
- gorilla/mux（路由）
- spf13/cobra（CLI）
- spf13/viper（配置）
- stretchr/testify（测试工具）
- sirupsen/logrus（日志）
- go-redis/redis（Redis 客户端）
- grpc/grpc-go（gRPC）
- prometheus/client_golang（监控）

**扫描命令**：
```bash
gosec -quiet -fmt=json -out=/tmp/gosec-<project>.json ./...
```

**环境**：
- gosec dev（Go 1.26.2）
- darwin/arm64（Apple M4 Pro）
- 扫描时间：~60 秒（全部项目）

**分类方法**（v2）：
- **兜底层相关（layer1）**：语言/运行时本可避免但主动开放给开发者的——`G103`（unsafe 包）、`G601`（range 指针）、`G504`（cgi）
- **开发者必守层（layer2）**：API 层安全——`G101` 硬编码凭证、`G107` SSRF、`G115` 整数溢出、`G20x` SQL 注入、`G304` Path Traversal、`G40x` crypto 误用等
- **代码卫生（hygiene）**：`G104`（未处理错误）——工程质量问题，不是严格安全漏洞，单独拎出

### HIGH 严重度 issues 分布（真正的安全问题）

| 分类 | 次数 | 占比 |
|------|:---:|:---:|
| **开发者必守层** | **220** | **100.0%** |
| 兜底层相关 | 0 | 0.0% |
| 代码卫生 | 0 | 0.0% |

**核心结论**：
- **gosec 识别出的 HIGH 严重度安全问题，100% 分布在开发者必守层**（220/220）
- 兜底层相关的发现（主要是 `unsafe` 包使用）严重度都是 LOW/MEDIUM——因为 Go 的设计就是这样：`unsafe` 是你主动选择进入的，不是语言强加的风险

## 证伪检查（重要）

**立意阶段假设**："80% 真实漏洞出在第二层"

**证伪结果**：
- 按总数算，第二层只占 22.8%（卫生类 67% 吞了大头）
- **按 HIGH 严重度算，第二层占 100%**——这是更精确的判定
- 立意阶段的判断**方向正确，但需要更精确的表达**：不是"总数 80%"，而是"真正高危的 100%"

**正文修改要点**：
- 不说"80% 事故出在第二层"（数据不支撑）
- 改说"在 gosec 扫出的 HIGH 严重度安全问题中，100% 出在开发者必守层"——**更锋利，数据更硬**
- 把 G104 分离作为代码卫生类，诚实呈现（避免冷读视角 2"统计口径不严谨"质疑）
