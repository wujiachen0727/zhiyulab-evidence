# 突发流量模拟实验

## 目的
模拟高并发请求突发涌入时，无限制创建 goroutine vs 协程池限流对系统资源的影响。

## 实验方法
创建 5000 个并发请求，每个请求处理约 10ms（模拟中等复杂度数据库查询），对比两种模式下的内存和 GC 压力。

## 运行环境
- Go 版本：1.26.4
- 操作系统：darwin/arm64 (macOS)
- CPU：Apple M4 Pro, 14 核

## 关键发现

### 短任务（1ms，10000 请求）
| 模式 | 耗时 | heapAlloc | heapSys | GC 次数 |
|:---:|:----:|:---------:|:-------:|:-------:|
| 直接 go func() | 26ms | 2.6MB | 22.1MB | 1 |
| 协程池 100 workers | 116ms | 1.9MB | 22.2MB | 3 |

### 长任务（10ms + 16KB 分配，5000 请求）
| 模式 | 耗时 | heapAlloc | heapSys | GC 次数 |
|:---:|:----:|:---------:|:-------:|:-------:|
| 直接 go func() | 30ms | 2.9MB | 26.2MB | 1 |
| 协程池 50 workers | 1095ms | 2.3MB | 26.5MB | 3 |

## 核心结论

**协程池的真正价值不是让任务更快，而是防止系统过载。** 从数据可以清晰看到：

1. **速度层面**：池总是更慢——因为 worker 数量限制导致任务排队。50 workers × 10ms = 每个 worker 串行处理，5000 个任务自然需要 ~1000ms。这不是池的"缺陷"，而是"限流"的必然结果。

2. **内存层面**：池在短任务场景下略有优势（2.6MB vs 1.9MB），差距不大。因为 goroutine 的栈（~4KB）相对于每个请求 16KB 的堆分配来说不是主要消耗。

3. **真正的差异不在微基准测试中体现**：在 micro-benchmark 里，5000 个 goroutine 创建 30ms 就完成，似乎"goroutine 确实很轻量"。但真实的线上场景是：
   - goroutine 不会 30ms 就结束——HTTP 请求、数据库查询通常 50-500ms
   - goroutine 数量不会刚好 5000 就停——持续流量下会累积到几十万
   - 不受限制的 goroutine 会让 GC 压力持续增长，最终 STW 时间失控

**结论**：不要用微基准测试的"快"来否定协程池的价值。协程池不是在优化性能，是在做资源管控。
