# Benchmark 对比实验

## 目的
量化"协程池 vs 直接 go func()"在 CPU 密集和 I/O 模拟场景下的性能差距。

## 实验方法
使用 Go 内置 benchmark 框架，在 10/100/1000/10000 四个任务量级下对比直接创建 goroutine 和使用 worker pool（8 个 worker）的性能。同时测试不同 worker 数量对吞吐的影响。

## 运行环境
- Go 版本：1.26.4
- 操作系统：darwin/arm64 (macOS)
- CPU：Apple M4 Pro, 14 核
- 测试命令：`go test -bench=. -benchmem -benchtime=1x`

## CPU 密集任务结果

### 直接 go func() vs 协程池

| 任务数 | 直接 go func() | 协程池 (8 workers) | 内存差距 |
|:-----:|:--------------:|:------------------:|:--------:|
| 10 | 66.7µs / 5.0KB / 22 allocs | 44.0µs / 4.8KB / 24 allocs | 基本持平 |
| 100 | 112µs / 55.9KB / 206 allocs | 101µs / 6.5KB / 17 allocs | 池内存低 **8.6x** |
| 1000 | 567µs / 288KB / 1501 allocs | 594µs / 8.5KB / 11 allocs | 池内存低 **34x** |
| 10000 | 4.9ms / 1.4MB / 12382 allocs | 6.1ms / 82KB / 13 allocs | 池内存低 **17x** |

### I/O 模拟任务结果

| 任务数 | 直接 go func() | 协程池 (8 workers) | 内存差距 |
|:-----:|:--------------:|:------------------:|:--------:|
| 10 | 271µs / 368B / 12 allocs | 273µs / 456B / 12 allocs | 基本持平 |
| 100 | 1.47ms / 2.4KB / 101 allocs | 2.26ms / 1.3KB / 12 allocs | 池内存低 **1.9x** |
| 1000 | 15.3ms / 24KB / 1002 allocs | 18.1ms / 8.6KB / 13 allocs | 池内存低 **2.8x** |
| 10000 | 125ms / 2.4MB / 14111 allocs | 163ms / 82KB / 11 allocs | 池内存低 **29x** |

### Worker 数量对性能影响（1000 CPU 任务）

| Workers | 耗时 | 内存 |
|:------:|:----:|:----:|
| 1 | 2.67ms | 8.3KB |
| 4 | 793µs | 8.4KB |
| 8 | 650µs | 8.5KB |
| 16 | 595µs | 9.3KB |
| 32 | 547µs | 8.8KB |

## 结论
1. **CPU 密集场景下，协程池最大优势不是速度，而是内存**：allocs 从 12382 降到 13，减少 952 倍。这对于 GC 压力的影响是巨大的。
2. **I/O 场景下优势更明显**：CPU 密集时 goroutine 是计算密集的，调度开销相对小。I/O 场景下 goroutine 会阻塞等待，直接 go func() 会产生大量休眠 goroutine，内存浪费更大。
3. **Worker 数量接近 CPU 核数时收益最大**：从 1→4→8 提升明显（2.67ms→793µs→650µs），再往上边际收益递减。
4. **小量级（≤10）任务时，池和直接创建无显著差异**：这也是"协程池是过度工程"观点的来源——但到了 1000+ 量级差距就出来了。
