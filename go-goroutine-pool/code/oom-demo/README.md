# OOM Demo 实验

## 目的
演示无限制创建 goroutine 对内存的影响，量化"不用池的代价"。

## 实验方法
创建一个受控程序，逐步创建 10 万个 goroutine，每个 goroutine 分配 4KB 堆内存，每批 1000 个记录一次内存状态。

## 运行环境
- Go 版本：1.26.4
- 操作系统：darwin/arm64 (macOS)
- CPU：14 核

## 关键数据

| goroutine 数量 | heapAlloc | heapSys | heapObjects | GC 次数 |
|:-------------:|:---------:|:-------:|:----------:|:-------:|
| 初始 | 0.24 MB | 3.72 MB | 189 | 0 |
| 10,000 | 6.57 MB | 11.84 MB | 30,355 | 1 |
| 50,000 | 30.41 MB | 37.59 MB | 124,000 | 4 |
| 100,000 | 61.89 MB | 71.94 MB | 259,617 | 5 |

## 结论
- 10 万个轻量 goroutine（每个仅 4KB 堆分配）就消耗了 ~62MB 堆内存，系统占用 ~72MB
- 外推：100 万个 goroutine → ~620MB 堆内存 + goroutine 栈（每个 ~4KB）→ 约 4GB 总内存，远超常规服务器配置
- OOM 临界点：约 50-80 万 goroutine（取决于服务器内存配置，4GB 内存的服务器）
- 注意：这只是"每个 goroutine 只做一件事"的场景。如果 goroutine 内有更多分配，临界点会大幅提前

## 文件说明
- `main.go`：无限制创建 goroutine 的原始版本（可能触发 OOM）
- `controlled.go`：受控版本，创建 10 万个后安全退出
