## E1：协议层吞吐与延迟对比

**目录**：`code/e1-protocol-bench/`
**结果**：`output/e1-result.md`

**结论**：
- 三种协议组合 P50 延迟相差极小（127µs ~ 404µs）
- HTTP/2 + 二进制 vs HTTP/2 + JSON 的 P50 差距：**1µs**——序列化形式对端到端延迟的影响近乎为零
- 二进制编码比 JSON 体积压缩到 **46.9%**——但响应大小本身就只有 179B，省下 95B 在 P50 维度根本看不出来
- 一个反直觉发现：**HTTP/1.1 + JSON 的 QPS 比 HTTP/2 还高**（10.5w vs 7.7w）。原因是同主机 loopback 下，HTTP/1.1 的连接池调度比 HTTP/2 单连接多路复用更轻——这也正是论点的佐证："快不快"取决于具体场景，不是协议本身

**复现命令**：
```bash
cd evidence/code/e1-protocol-bench
go build && ./e1-protocol-bench
```

## E2：端到端调用延迟分解

**目录**：`code/e2-latency-breakdown/`
**结果**：`output/e2-result.md`

**结论**：业务逻辑模拟 10ms（代表一次中等量级 DB/下游调用）：

| 环节 | 占比 |
|------|:----:|
| 请求序列化（JSON Marshal） | 0.02% |
| 网络栈 + 协议处理 | 15.96% |
| **业务逻辑** | **83.64%** |
| 响应反序列化（JSON Unmarshal） | 0.20% |

**敏感性分析**——业务越重，协议层占比越小：
- 业务 10ms → 协议+序列化占比 16.19%
- 业务 20ms → 8.82%
- 业务 50ms → 3.73%
- 业务 100ms → 1.90%

**关键推论**：在真实生产环境（业务通常 ≥ 20ms），协议优化的天花板上限是个位数百分比。

**复现命令**：
```bash
cd evidence/code/e2-latency-breakdown
go build && ./e2-latency-breakdown
```
