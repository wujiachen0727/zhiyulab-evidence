> 基于 E1+E4 / E2 / E3 / E5 的实测数据提炼。
> 目标：给读者一张可行动的"我该用哪一层工具"决策表。

## 三层工具链能力矩阵

| 能力维度 | pprof（第 1 层）| trace（第 2 层）| 持续 profiling（第 3 层）|
|:--------|:-------------:|:-------------:|:----------------------:|
| CPU 热点（哪个函数占多少 CPU）| ✅ 但硬编码 100Hz | ❌ 不直接给 | ✅ 时间序列的 CPU 热点 |
| 内存/堆分配热点 | ✅ | ❌ | ✅ |
| **锁等待累计时长** | ✅ 但归因反直觉（E2） | ✅ | ✅ |
| **等待的时间序列** | ❌ | ✅ | ❌ |
| **阻塞发生的次数** | ❌ | ✅（E2：11 万次）| ❌ |
| **goroutine 调度关系** | ❌ | ✅ | ❌ |
| **runnable 但抢不到 CPU** | ❌ | ✅（E2：0.99s） | ❌ |
| **GC STW 事件详情** | ❌ | ✅ | ❌ |
| **syscall 阻塞事件** | ❌ | ✅ | ❌ |
| **毛刺的具体时间点** | ❌（E3：被稀释到底噪）| ❌（默认 < 10 秒采样）| ✅（E3：时间序列能定位）|
| **历史 profile 对比** | ❌（每次独立）| ❌ | ✅ |
| **跨节点/跨进程聚合** | ❌ | ❌ | ✅ |
| **版本/发布 diff** | ❌ | ❌ | ✅ |
| **生产长期开启** | ⚠️ 可以但要外部调度 | ❌ 开销大 | ✅ 1-4% 开销可控（E5）|

## 升级决策表（按症状）

### 决策树

```
性能问题来了——

第 1 步：症状是 CPU 密集吗？
├── 是 → pprof CPU profile 就够了
└── 不是 → 进入第 2 步

第 2 步：涉及等待吗？（锁、channel、IO、syscall）
├── 是 → pprof block/mutex profile 能看聚合 → 
│       还需要看时间序列/因果 → trace
│       （block/mutex profile 足够的场景：累计量大就是问题）
│       （需要 trace 的场景：高频阻塞、goroutine 之间因果）
└── 不是 → 进入第 3 步

第 3 步：问题是偶发的、历史回溯的、跨节点的吗？
├── 是 → 持续 profiling（Pyroscope / Parca / Grafana Phlare）
└── 不是 → pprof + 短暂 trace 就够了
```

### 场景 → 工具对照表

| 症状 | 首选工具 | 为什么 | 实验依据 |
|:-----|:--------:|:------|:---------|
| 服务 CPU 使用率高 | **pprof CPU profile** | 直接看热点函数 | E1+E4 |
| 服务 QPS 上不去但 CPU 不满 | **trace** | 大概率等待问题，pprof CPU profile 看不见 | E2 |
| P99 延迟高但 P50 正常 | **trace**（短时）| 需要看单次请求的调度路径 | E2 |
| GC STW 疑似过长 | **trace** | 只有 trace 能给 STW 事件详情 | 官方 |
| 内存持续增长 | **pprof heap profile** + **持续 profiling** | heap 给快照，持续给增长曲线 | — |
| 偶发延迟毛刺 | **持续 profiling** | pprof 采样窗口会稀释毛刺（E3 实测 4.80% vs 16.47%）| E3 |
| 版本发布后性能下降 | **持续 profiling**（做 diff）| 需要和上版本对比 | E5 推演 |
| 某个 pod 慢，其他正常 | **持续 profiling**（多节点聚合）| 需要跨节点对比 | E5 推演 |
| 一次性调查特定请求 | **go test -bench + pprof** | 最精确的环境 | — |
| goroutine 泄漏 | **pprof goroutine profile** | 专门 profile | — |

## 升级决策的三个"触发信号"

### 信号 1：pprof 看不见问题 → 升级到 trace

**触发条件**（满足任一）：
- 服务慢但 CPU profile 的 cum 只有 runtime 调度器（E2 实测：50 worker 锁争用 CPU profile 业务代码 0%）
- 需要知道"等了多少次"而不是"等了多久"
- 需要 goroutine 之间的因果关系（谁在等谁）
- 需要看 GC STW 事件详情
- 需要看"runnable 但抢不到 CPU"

**升级成本**：trace 的开销比 pprof 大（Go 官方数据 2-10% CPU），通常只能短期跑（几秒-几分钟）。

### 信号 2：问题不在"此刻"而在"某个时段" → 升级到持续 profiling

**触发条件**（满足任一）：
- 问题是偶发的（E3 实测：30s pprof 窗口把毛刺稀释到 4.80%）
- 需要历史回溯（比如"昨天下午 3 点那次抖动")
- 需要版本间对比（发布前后 diff）
- 需要跨节点聚合（哪个 pod 异常）
- 不是救火，是**体检**——你想长期监控，不是出问题才打开

**升级成本**（E5 实测）：
- QPS 损失约 1.61%
- 进程 CPU +3.70%
- 进程 RSS +7.8MB
- 运维：需要维护 Pyroscope/Parca 后端 + 客户端 SDK

### 信号 3：pprof/trace 够用，不要升级

**满足条件**（推荐继续只用 pprof）：
- 服务规模小（< 1000 QPS）
- 问题已知且能复现
- 团队还在入门阶段，多一个工具就是多一个复杂度
- **流量本身就是瓶颈**（代理、CDN 边缘节点）——为每 0.5% QPS 斗的场景

## 常见错配（别这么干）

| 错配 | 问题 | 正确做法 |
|:-----|:-----|:---------|
| 拿 pprof 查锁等待 | CPU profile 业务代码 0%，block profile 给聚合不给结构 | 直接上 trace |
| 拿 trace 长期监控 | 开销大（2-10%），文件大（MB 级），只能短期 | 用持续 profiling |
| 生产开 trace 长期跑 | trace 缓冲区会撑爆磁盘 | 触发式——只在问题复现时开 |
| 同时开 pprof + Pyroscope agent | Go CPU profile 是全局单例（E3/E5 发现），会冲突 | 装了 agent 后，`go tool pprof` 请求会偶发失败；要么停 agent 要么用 Pyroscope 的 UI |
| 只看 pprof 最宽的柱子 | 最宽 ≠ 瓶颈（E1+E4：长函数被异步抢占低估 15%，E2：等待根本不出现在 CPU profile）| 先确认"瓶颈是 CPU 密集还是等待"，再选工具 |

## 给团队的落地建议

### 入门阶段（Go 服务规模 < 100 QPS）
- 只用 pprof。先能读懂火焰图
- 学会：CPU profile / heap profile / goroutine profile 三件套

### 成长阶段（100-10000 QPS，开始出现偶发问题）
- 加入 trace。学会看时间轴、调度、阻塞
- 用 `net/http/pprof` 按需抓取

### 成熟阶段（> 10000 QPS，生产稳定性重要）
- 部署持续 profiling（Pyroscope / Parca）
- 接入 Grafana，和指标/日志/trace 联动
- 建立"发布后对比"流程——每次发布用 diff 功能找性能回归

## 这份决策表的独特性

市面上大多数 Go 性能文章：
- 只讲"pprof 怎么用"——不讲"什么时候不够用"
- 只讲"trace 是什么"——不讲"什么时候该升级"
- 只讲"Pyroscope 能做什么"——不讲"什么时候不该用"

本表基于 **49 份本地实测数据**，给出**触发信号 + 升级成本 + 常见错配 + 团队阶段建议**的四维决策框架——这是中文语境几乎没有的角度。
