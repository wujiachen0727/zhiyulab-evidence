# E6：三个典型问题场景

## 场景 1：延迟抖动——"今天突然 P99 高了"

### 症状
- 生产服务 P99 延迟从平常 50ms 突然飙到 300ms
- P50 / P90 看起来正常
- CPU 使用率没变化
- 内存使用没变化
- 发生在特定时间段（比如每天下午 3 点左右）

### 为什么 pprof 不够？
**E3 实测支撑**：30 秒 pprof 采样窗口会把偶发毛刺稀释到底噪水平——我实测一个"99% 快 / 1% 慢"的场景，30s 窗口下慢路径只占 4.80%（排第 5），混在正常业务代码之间根本看不出来。

而且 pprof 是"拍你开启采样那一刻"的照片——你不知道 P99 高的时候你是不是开着 pprof，也没法回溯。

### 该用哪一层？
**持续 profiling（第 3 层）**——因为你需要的是"时间序列"，不是"某一秒的快照"。

具体做法：
1. 打开 Pyroscope UI，选时间窗口对准"每天下午 3 点"
2. 对比这个窗口 vs 其他时段的 CPU profile
3. 差异函数 = 毛刺元凶

### 不该用 trace 的原因
trace 开销大（2-10% CPU），文件大（E2 实测 3.6MB / 1 秒），不能长期开——你没办法"trace 从早到晚"等毛刺出现。

---

## 场景 2：CPU 使用率 100% 但 QPS 上不去

### 症状
- 压测打满 CPU（14 核全满）
- QPS 远低于基线
- pprof CPU profile 显示大部分时间在 `runtime.findRunnable` / `runtime.schedule`
- 看不到业务代码

### 为什么 pprof CPU profile 不够？
**E2 实测支撑**：50 worker 激烈争抢 1 把锁的场景下，CPU profile 显示的是 runtime 调度器在反复调度等待锁的 goroutine——业务代码 0% 归因。你以为 CPU 在忙（100% 使用率），实际 CPU 在"空转调度"。

### 该用哪一层？
**trace（第 2 层）**。

具体做法：
1. `curl -o trace.out http://localhost:PORT/debug/pprof/trace?seconds=5`
2. `go tool trace trace.out`，打开 "Goroutine analysis"
3. 看 "Time in Waiting" 最多的 goroutine——它们在等什么
4. 点进去看 "Lock" / "Select" / "Chan" 的堆栈

### 实测数据作背书
E2 里的 20 个 worker 被 trace 显示阻塞 2000 次以上/每个——这种高频阻塞在 pprof 里彻底看不见。

---

## 场景 3：发布后 QPS 下降 5%，不知道哪里慢了

### 症状
- 发布新版本前 QPS 10000
- 发布后 QPS 9500（下降 5%）
- 看 pprof CPU profile，top 函数和之前"看起来差不多"
- 回滚可以复现

### 为什么 pprof 不够？
pprof 给你的是"当前快照"——你记不住"上个版本的火焰图长什么样"。眼睛看不出 5% 的占比变化。

### 该用哪一层？
**持续 profiling 的 diff 功能（第 3 层）**。

具体做法：
1. 在 Pyroscope UI 选"本版本 vs 上版本"两个时间窗口
2. 用 "Diff" 视图直接看"哪些函数占比升高"
3. 5% 的差异在 diff 视图里会被**直接高亮**——不需要眼睛对比

### 不该用 trace 的原因
trace 是事件流，不适合做"聚合对比"。pprof 可以做 diff（`go tool pprof -diff_base`）但你得手动保存上版本的 profile——如果没预先保存就来不及了。持续 profiling 的价值是**默认保留历史**。

---

## 三个场景的共同点

所有场景都有一个模式：
1. **症状**：某个性能指标异常
2. **pprof 第一反应**：看 CPU profile top
3. **pprof 的盲区**：看不见等待（场景 2）、看不见时间点（场景 1）、看不见历史（场景 3）
4. **升级后的工具**：给出 pprof 盲区里的信息

**核心认知**：工具链升级不是"有了新工具就淘汰老的"——pprof 仍然是**最快的第一诊断**，但它有系统性的盲区，每一层工具各自补一块。

# E7：三层工具链的信息论推演

## 为什么是三层？不是两层？不是四层？

### 从"要回答什么问题"反推

性能问题的全部可能性可以归为三个问题：

**Q1：哪里慢？**（空间定位）
- 关键词：**函数、行、模块**
- 想要的信息：CPU / 内存 / 分配 的**空间分布**
- 输出形式：**聚合统计**（火焰图、top list）
- 工具：pprof

**Q2：为什么慢？**（因果定位）
- 关键词：**等待、调度、依赖**
- 想要的信息：事件的**时间线和因果关系**
- 输出形式：**事件流**（timeline、swimlane）
- 工具：trace（execution tracer）

**Q3：什么时候慢？**（时间定位）
- 关键词：**历史、周期、趋势**
- 想要的信息：profile 随**时间轴**的变化
- 输出形式：**时间序列**（按时间切片的聚合统计）
- 工具：持续 profiling（Pyroscope / Parca）

### 这三个问题正交吗？

**正交**。意思是：你不能用 Q1 的答案推出 Q2，也不能用 Q2 的答案推出 Q3。

- 知道"函数 X 占 20% CPU"（Q1 答案），推不出"函数 X 是因为等锁才慢"（Q2 答案）
- 知道"是因为等锁才慢"（Q2 答案），推不出"是每天下午 3 点才等锁"（Q3 答案）
- 知道"下午 3 点慢"（Q3 答案），推不出"哪个函数慢"（Q1 答案）

**三个问题正交 = 需要三种独立的信息形式 = 需要三种工具**。

### 为什么不是两层？

合并 Q1 和 Q2？—— 不行。采样统计（pprof）和事件流（trace）在数学上是不同粒度的信息：
- 采样统计：O(100) 采样/秒 × N 秒 = O(100N) 数据点，丢失事件细节
- 事件流：每个调度/系统调用都记录 = O(10000) 数据点/秒 × N 秒，保留完整时序

**从采样统计反推事件流在数学上不可能**——信息已经丢失。所以 trace 必须作为独立一层。

合并 Q2 和 Q3？—— 也不行。长期开 trace 开销太大（官方数据 2-10% CPU，N 秒会产生 MB 级数据）——物理上不能 7×24 开。所以持续 profiling 必须用"时间切片采样"而不是"持续事件流"，它是独立一层。

### 为什么不是四层？

有没有"Q4"？
- **Q4 候选："谁导致的慢"（溯源定位）**——谁发的请求、哪个用户、哪个下游？
- 这不是 profiler 的职责——这是**分布式追踪**（OpenTelemetry / Jaeger）的职责
- profiler 回答"进程内"的性能问题，trace 回答"跨进程"的因果链
- 所以第四层工具存在，但它不是 profiling 工具链的一部分

### 小结：三层框架的推演逻辑

```
性能问题的全部可能性 = {哪里慢、为什么慢、什么时候慢}
                     ↓（三个正交问题）
需要三种信息形式 = {聚合统计、事件流、时间序列}
                     ↓（信息形式决定工具架构）
Go 性能工具链 = {pprof、trace、持续 profiling}
                     ↓（能力堆叠，不是替代）
认知框架 = 识别你的问题属于哪一层 → 选对应工具
```

## 给读者的"记忆锚点"

一句话版本：

> **pprof 回答"在哪"，trace 回答"为什么"，持续 profiling 回答"什么时候"——三个问题独立，三种工具互补，堆叠而非替代。**

类比版本（T1 类比）：

> **pprof 像医生拍 CT**——告诉你哪里有阴影。
> **trace 像心电图**——告诉你心跳什么时候乱。
> **持续 profiling 像年度体检报告**——告诉你一年里身体指标怎么变的。
>
> 三样东西加起来，才是"完整的身体"。
