# 双语 GC 基准测试结果

**测试环境**：macOS Darwin arm64 (Apple Silicon), 14 cores
**工作负载**：1000 万次 64B 对象分配，保持 10000 个活跃对象（模拟工作集）

## 结果汇总

| GC | 版本 | 总耗时 | GC 次数 | GC 总暂停 | 平均暂停 | 最大暂停 | 堆占用 |
|:---|:-----|:------:|:-------:|:---------:|:-------:|:-------:|:------:|
| **Go** (GOGC=100) | Go 1.26.2 | 131ms | 225 | 8.6ms | 38µs | 158µs | 3.3MB |
| **Java G1** (-Xmx256m) | JDK 21.0.10 | 47ms | 14 | 3ms | ~214µs | — | 23MB |
| **Java ZGC** (-Xmx256m) | JDK 21.0.10 | 79ms | 12 | 9ms(cycles) / 0ms(pause) | ~0µs(STW) | — | 88MB |
| **Java Serial** (-Xmx256m) | JDK 21.0.10 | 37ms | 11 | 2ms | ~182µs | — | 17MB |
| **Java G1** (默认堆) | JDK 21.0.10 | 64ms | 7 | 2ms | ~286µs | — | 186MB |
| **Java ZGC** (默认堆) | JDK 21.0.10 | 115ms | 0(MXBean) | 0ms(pause) | ~0µs | — | 770MB |

## 关键发现

### 1. Go：频繁回收，极低暂停
- 225 次 GC，但每次暂停仅 38µs（最大 158µs）
- GC CPU 占比 3.45%
- 堆占用极小（3.3MB），因为不分代、频繁回收

### 2. Java G1：少量回收，中等暂停
- 14 次 GC（256m堆），每次暂停约 214µs
- 分代收集高效：大部分对象在 Young Generation 就被回收
- 堆占用 23MB——比 Go 大，但远低于 ZGC

### 3. Java ZGC：几乎零 STW，但吃内存
- ZGC Pauses 报告 0ms STW（亚毫秒，MXBean 精度不够）
- 代价：堆占用 88MB（256m限制）/ 770MB（默认堆）
- ZGC 用内存换延迟——和 Go 的策略相反

### 4. 吞吐量 vs 延迟的经典取舍
- Java Serial（STW 全暂停）吞吐最高：37ms
- Go（并发标记）吞吐最低：131ms——为极低暂停付出了 CPU 代价
- Java G1/ZGC 在中间

## 对文章的启示

这组数据完美支撑"同一场战争的两条路"叙事：
- **Go 的路**：频繁回收（225次 vs Java 的 7-14次），每次极短（38µs），代价是 CPU（3.45% GC 开销）
- **Java 的路**：分代+大堆少回收，但 ZGC 为了追平 Go 的低暂停，吃掉了 770MB 内存
- **核心结论**：Go 用 CPU 换确定性延迟，Java 用内存+复杂度换灵活性
