# SSA 优化链断裂实验结果

[实测 Go 1.26.2 darwin/arm64 Apple M4 Pro]

## 实验设计

验证核心论点：**编译器优化是链式传导的，内联断裂会导致后续优化全部失效。**

三组对照：
- **优化链完整**：小函数 `addInline` 被内联 → 编译器看到完整上下文 → 后续优化生效
- **优化链断裂**：同一逻辑通过接口调用 → 内联被阻断 → 后续优化链式失效
- **纯计算基准**：直接写 `i + (i+1)`，代表"编译器最理想情况"

## 编译器决策

| 场景 | 内联 | 去虚化 | 逃逸 |
|------|:----:|:------:|:----:|
| 优化链完整 | ✅ `inlining call to addInline` | N/A | ✅ 无逃逸 |
| 优化链断裂 | ❌ 接口调用未内联 | ❌ 动态分发 | ❌ `inlineAdder{}` escapes to heap |
| 纯计算基准 | N/A | N/A | ✅ 无逃逸 |

## Benchmark 结果

| 场景 | ns/op | B/op | allocs/op | vs 基准 |
|------|------:|-----:|----------:|:-------:|
| 纯直接计算 | ~485 | 0 | 0 | 1x |
| 优化链完整（内联生效） | ~488 | 0 | 0 | ~1x |
| 优化链断裂（内联被阻断） | ~1043 | 0 | 0 | ~2.1x |

## 关键发现

1. **内联让函数调用"消失"**：优化链完整版 ≈ 纯计算（488ns vs 485ns），内联后编译器生成的代码与手写直接计算几乎一致。
2. **内联断裂 = 链式失效**：同样的加法逻辑，优化链断裂版慢 2.1 倍。内联被阻断 → 编译器看不到函数体 → 逃逸分析受限 → 无法进一步优化。
3. **逃逸也在链上**：优化链完整版无逃逸，断裂版 `inlineAdder{}` 逃逸到堆上——接口封装导致额外的堆分配。
4. **这不是单个优化的失效，是整条链的断裂**：从内联→逃逸→死代码消除，一环断全链断。

## 实验环境

- Go 版本：1.26.2
- 平台：darwin/arm64
- CPU：Apple M4 Pro
