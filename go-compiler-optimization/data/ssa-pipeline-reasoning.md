# SSA Pipeline 推演：优化链的理论基础

[推演] 基于Go官方仓库 cmd/compile/internal/ssa/compile.go 源码

## 核心发现

**Inlining 和 Escape Analysis 不在 SSA pass 列表中**——它们在 SSA 构建之前由编译器前端（gc 包）完成。SSA 的入口 `Compile(f)` 接收的已经是经过内联和逃逸分析后的函数。

因此，Go 编译器的实际优化顺序是：

```
1. 内联（前端阶段）
   ↓ 内联展开函数体，提供上下文
2. 逃逸分析（前端阶段）
   ↓ 依赖内联后的上下文判断变量是否逃逸
3. SSA 构建 + SSA Pass 序列
   ├─ opt（主优化，含常量折叠、简化）
   ├─ generic cse（公共子表达式消除）
   ├─ nilcheckelim（空指针检查消除）
   ├─ prove（边界检查消除/BCE）
   ├─ deadcode ×7（死代码消除，贯穿全程）
   ├─ lower（降低为机器指令）
   └─ regalloc（寄存器分配）
```

## 优化链传导机制

### 链条1：内联 → 逃逸分析

**前提**：内联展开函数体后，编译器拥有更大的上下文
**推理**：
1. 函数未内联时，编译器只能按函数签名做保守逃逸分析
2. 函数内联后，编译器看到调用点的完整上下文
3. 在某些情况下，内联后可以证明变量不逃逸（如返回指针的函数被内联后，指针可能仅在被调用方使用）
4. **实测证据**：E2 实验中，未加 `//go:noinline` 时，所有逃逸的变量都变成了不逃逸——因为内联消除了函数边界，逃逸分析看到变量实际没逃出

**推理链**：内联失败 → 上下文丢失 → 逃逸分析保守 → 变量不必要地分配到堆上

### 链条2：内联 → 去虚化 → 后续优化

**前提**：接口方法的去虚化依赖编译器确定具体类型
**推理**：
1. 编译器需要看到接口变量的具体赋值才能去虚化
2. 如果方法被内联，编译器有更多上下文确定具体类型
3. 去虚化后，接口调用变为直接调用，可以进一步内联
4. **实测证据**：E3 实验中，`var p Processor = fastProcessor{}` 的调用被去虚化+内联，性能与直接调用一致

**推理链**：内联失败 → 去虚化失败 → 接口调用无法优化 → 动态分发开销

### 链条3：CSE → prove → deadcode

**前提**：从 SSA pass 顺序和 passOrder 约束
**推理**：
1. `generic cse` 消除公共子表达式，简化 IR
2. `prove` 依赖 `generic cse` 的结果（官方源码约束：`generic cse → prove`）
3. `prove` 消除边界检查后，`generic deadcode` 清理新产生的死代码
4. **约束链**：CSE 先行 → prove 才能有效推理 → deadcode 才能清理

**推理链**：CSE 失效 → prove 推理空间缩小 → BCE 消除不彻底 → 死代码残留

## "一环断全链断"的实验支撑

| 环节断裂 | 连锁效应 | 实验证据 |
|---------|---------|---------|
| 内联被阻断 | 逃逸分析保守 → 不必要堆分配 | E2：内联版 0 allocs vs noinline版 1 allocs |
| 内联被阻断 | 去虚化失败 → 动态分发 | E3：去虚化 ~259ns vs 未去虚化 ~1273ns |
| 内联被阻断 | 整体优化链失效 | E4：优化链完整 ~488ns vs 断裂 ~1043ns |

## 推演的前提声明

1. 内联和逃逸分析在 SSA 前端完成——来自 Go 官方源码注释
2. SSA pass 顺序和依赖——来自 cmd/compile/internal/ssa/compile.go 的 passOrder 约束
3. "内联为逃逸分析提供上下文"——基于 E2 实验数据的推理，非 Go 官方文档直接声明
4. 编译器行为随版本变化——以上基于 Go 1.26.2，未来版本可能不同
