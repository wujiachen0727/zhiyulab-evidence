# keepalive / idle timeout 配置组合对比

标注：本表为 [推演]，用于解释不同配置组合的时间关系；Linux 默认参数来自 tcp(7) 文档事实，云厂商 idle timeout 来自立意阶段求证快照。

| 场景 | 中间设备 idle timeout | TCP 首次 keepalive | TCP 最晚判死时间 | 连接池 idle timeout | 判定 | 说明 |
|---|---:|---:|---:|---:|---|---|
| Linux 默认 keepalive + 5 分钟级 NAT | 300s | 7200s | 7875s | 未设置 | 探测晚于回收：存在陈旧连接窗口 | Linux tcp(7) 默认参数；SO_KEEPALIVE 开启后仍要等 2 小时才首次探测 |
| AWS NAT Gateway 350s + Linux 默认 keepalive | 350s | 7200s | 7875s | 未设置 | 探测晚于回收：存在陈旧连接窗口 | AWS 文档示例；350s 仍远早于 7200s |
| Azure LB 默认 240s + Linux 默认 keepalive | 240s | 7200s | 7875s | 未设置 | 探测晚于回收：存在陈旧连接窗口 | Azure 默认 4 分钟；不能把 5 分钟写成通用默认值 |
| 调低 TCP keepalive：60/10/3 | 300s | 60s | 90s | 未设置 | TCP 探测早于中间设备回收 | 首次探测早于 300s，最晚约 90s 判死；适合需要长连接保活的场景 |
| 连接池 idle timeout 240s | 300s | 7200s | 7875s | 240s | 连接池先回收：安全边界更靠前 | 不依赖 TCP 探活，应用侧先丢弃旧连接 |
| 错误调参：keepalive_idle=600s | 300s | 600s | 690s | 未设置 | 探测晚于回收：存在陈旧连接窗口 | 看似调低了 2 小时，但仍晚于 5 分钟级回收 |

## 可供正文引用的结论

1. 如果中间设备 300s 左右回收空闲连接，而 TCP keepalive 7200s 才首次探测，那么 TCP 探测一定来得太晚。
2. 修复原则不是固定改成某个数字，而是让主动探测或连接池回收发生在中间设备回收之前。
3. 只把 keepalive 从 7200s 改成 600s 仍可能无效，因为它依然晚于 300s 的回收窗口。
