# 接口超时排查决策表

## 证据性质

[推演]

本表不是某次生产事故的完整复盘记录，而是把本文的排查逻辑整理成可迁移决策树。正文使用时应写成“可以按这个顺序查”，不要伪装成用户真实现场步骤。

## 决策表

| 看到的信号 | 优先怀疑 | 下一步检查 | 能排除什么 |
|---|---|---|---|
| 下游接口耗时整体升高，所有连接都慢 | 下游服务慢或网络整体拥塞 | 查下游 SLO、服务端日志、链路追踪 | 排除单条旧连接问题 |
| 只有复用连接的首次请求偶发失败，重试后恢复 | 陈旧连接 / 中间设备回收 | 查连接池 idle timeout、max lifetime、是否启用健康检查 | 排除“接口一直慢” |
| 错误集中出现在长时间空闲后 | idle timeout 错配 | 对齐业务低峰、连接空闲时间、NAT/LB idle timeout | 定位回收窗口 |
| 抓包看到 RST | 中间设备显式重置或对端关闭 | 判断 RST 来源方向，结合 NAT/LB 文档 | 区分黑洞 timeout 与显式重置 |
| 抓包只看到重传直到应用超时 | 链路黑洞或状态不一致 | 查安全组/NAT/LB 状态、长连接保活 | 解释为什么应用层才感知 |
| sysctl 显示 keepalive 首次探测远晚于 idle timeout | TCP 探活来得太晚 | 调整 socket keepalive 或连接池生命周期 | 避免只加大业务 timeout |

## 章节使用建议

- 第 3 章：用作“从 timeout 日志到内核参数”的排查路径表。
- 第 4 章：结合概念拆分解释“应用 timeout、连接池 idle timeout、TCP keepalive、HTTP keep-alive”分别解决什么问题。
- 第 5 章：收束成最小修复原则：探测早于回收，失败早于用户感知。

## 可供正文引用的结论

1. 如果失败只发生在长时间空闲后的第一次复用连接，排查重点应从“接口慢”转向“旧连接是否还活着”。
2. 如果中间设备 idle timeout 早于 TCP keepalive，单纯拉长应用 timeout 只会让用户等更久，不会让连接更早变健康。
3. 真正有效的策略通常是二选一或组合：应用侧更早丢弃旧连接，或 TCP/应用层更早探活。
