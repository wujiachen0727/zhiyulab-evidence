# E1 vs E2 结构对比数据

> 基于同一个用户管理 CRUD 场景（5 个 API：List/Get/Create/Update/Delete）

## 基础指标

| 指标 | E1（无分层） | E2（接口分层） | 差异 |
|------|:-----------:|:------------:|------|
| Go 文件数 | 1 | 6（不含测试） | +5 |
| 代码行数（不含测试） | 190 | 325 | +71% |
| 代码行数（含测试） | 190 | 417 | +120% |
| 目录层级 | 1 层（根目录） | 4 层（handler/service/repository/model） | +3 |

## "换数据源"改动量对比

**场景：从 MySQL 直连改为加一层 Redis 缓存**

| 需要改动的地方 | E1（无分层） | E2（接口分层） |
|--------------|:-----------:|:------------:|
| 涉及文件数 | **5 个 handler 函数全部要改**（散落在 main.go） | **1 个文件**（新增 redis_user.go 实现接口）+ main.go 换注入 |
| 改动点 | 每个 handler 里的 SQL 语句逐一替换为 Redis 调用 | 新写一个 `CachedUserRepo` 实现 `UserRepository` 接口 |
| 风险 | 改一个漏一个；SQL 和 Redis 调用混杂在 HTTP 处理逻辑中 | 接口不变，service 和 handler 零改动 |
| 回滚方案 | 逐行还原 | main.go 里把注入换回 `MySQLUserRepo` |

## "写单测"难度对比

**场景：给 CreateUser 写单元测试**

| 维度 | E1（无分层） | E2（接口分层） |
|------|:-----------:|:------------:|
| 能否不启动数据库 | 不能。handler 直接调用全局 `db`，必须有真实数据库或 sqlmock | 可以。service 依赖 `UserRepository` 接口，手写 mock 即可 |
| mock 方式 | 需要 `go-sqlmock` 库 mock SQL 层，或启动测试数据库 | 手写一个 struct 实现接口，零依赖 |
| 测试隔离 | 全局变量 `db` 被所有测试共享，并行测试需要额外处理 | 每个测试函数创建独立的 mock，天然隔离 |
| 测试速度 | 取决于数据库连接速度（通常 100ms+ 级别） | 纯内存操作（通常 < 1ms） |
| 典型测试代码行数 | ~40 行（含 sqlmock 设置） | ~15 行（见 user_service_test.go） |

## 关键洞察

**E1 不是"错误"的写法** —— 它是最快启动的写法。问题在于它的改动成本随项目规模线性增长：

- 5 个 API 时：一个人改一个文件，还挺快
- 50 个 API 时：SQL 散落 50 个函数里，改一次数据源是噩梦
- 加测试时：要么引入 sqlmock（学习成本），要么启动测试数据库（环境成本）

**E2 的前期投入**（多 71% 的代码量）换来的是：
- 换数据源：改 1 个文件 vs 改 N 个 handler
- 写测试：手写 mock 15 行 vs sqlmock 40 行 + 数据库
- 新人阅读：每个文件职责单一，看文件名就知道去哪找
